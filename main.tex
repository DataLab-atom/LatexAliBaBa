\section{项目介绍}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{项目标题}

\textbf{基于大模型冷启动蒸馏的 MCP 端点专家模型自动化生产——面向 AIOps 故障应急响应}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{背景与意义}

\textbf{产业背景。}
阿里巴巴等大规模互联网平台运维着复杂异构的基础设施，须保障近乎零停机时间。故障应急响应（AIOps）是关键运维能力，但目前仍高度依赖人工：值班工程师须在严峻时间压力下诊断告警、跨数十子系统关联日志并给出修复方案。
Qwen3 系列模型 \citep{qwen3} 及 LLM（大语言模型）的整体成熟，为该流程的自动化打开了大门。
然而，单次云端 LLM API 调用延迟高达 500\,ms--2\,s，在大规模故障风暴下甚至会超时失败；
而本地部署的 SLM（小语言模型）响应时延 $<$100\,ms，对延迟敏感的 AIOps 执行层不可或缺。

\textbf{LLM–SLM 协同研究中的隐含前提。}
课题一要求探索大小模型如何有效协同。现有协同框架（如 FrugalGPT、RouteLLM）隐含假设
\emph{专家小模型已经存在}：其贡献在于高效的路由或调度策略，决定由哪个模型处理某次查询。
然而，这些工作并未回答更根本的问题：\emph{专家小模型从何而来？}

\textbf{推理链训练是构建高能力 SLM 的成熟路径。}
另一研究方向——STaR \citep{star}、Distilling Step-by-Step \citep{dss} 及相关方法
\citep{minillm,ddk,onpolicy}——已充分证明：在训练时为 SLM 引入\emph{思维链（CoT）推理}，
而非仅记忆输入输出对，能显著提升其在领域特定任务上的性能。
瓶颈在于数据：获取高质量 CoT 标注，需要昂贵的人工标注员或能按需生成推理链的大型教师模型。
在\emph{冷启动}场景下——即团队进入无标注数据、无现成专家模型的新业务领域时——上述资源均不可得。

\textbf{研究空白。}
如图~\ref{fig:gap} 所示，现有 SLM 生产方法均依赖以下至少一项：
(a) 人工标注 CoT；(b) 具备领域知识的教师模型；(c) 可用于引导的已有学生模型。
而在冷启动场景——恰恰是新 AIOps 子场景的常态——上述前提均不成立。
这一空白激发了本项目的核心提案：一套\emph{仅从原始 $\langle$输入, 输出$\rangle$ 对}
自动生成 CoT 标注训练数据的流水线，无需任何人工介入，也无需具备领域知识的教师模型。

\textbf{现有问题与困难。}

\textbf{（1）冷启动场景下专家 SLM 生产"无自动化路径"。}
现有 LLM--SLM 协同框架将专家小模型视为已知前提：FrugalGPT、RouteLLM 等工作贡献
路由或调度策略，但不提供 Executor 本身的生产方法。
实践中，新 AIOps 子场景（如新引入的服务网格或新型微服务故障类型）初始时既无标注数据，
也无预训练领域模型。若缺乏从零自动化生产专家 SLM 的手段，整个协同技术栈将无 Executor 可路由。
如何构建一套完全自动化、无需标注的 SLM 生产流水线，仅从原始运维记录出发，是第一项核心挑战。

\textbf{（2）工业领域的答案验证 CoT 合成"仍未解决"。}
基于 CoT 的蒸馏方法——STaR \citep{star}、Distilling Step-by-Step \citep{dss}——已充分
证明推理链训练相比标准输入输出微调能产出更强的 SLM。然而，生成\emph{经验证}的 CoT 标注，
需要判断每条推理链是否导向正确答案，这又依赖人工评估者或领域专家教师模型。
冷启动时两者均不存在。如何仅凭历史记录中已有的标准答案，在无任何领域特定监督的情况下
自动验证候选推理链，且能泛化到工业 AIOps 各子场景，是第二项核心挑战。

本项目通过 MOLE1 流水线同时攻克上述两个问题，以阿里巴巴 AIOps 运营数据作为主要验证环境。

\begin{figure}[H]
\centering
\input{fig_gap.tikz}
\caption{现有 SLM 生产方法在冷启动场景下失效（左）；MOLE1 以观测答案作为自动验证器，突破这一瓶颈（右）。}
\label{fig:gap}
\end{figure}

\textbf{为何 AIOps 是理想验证场景。}
三项结构性特点使 AIOps 成为验证本方案的最佳领域。
\begin{enumerate}[leftmargin=*, itemsep=2pt]
  \item \textbf{数据丰富，无标注成本。}
        每个大规模平台的运维数据库中均积累了大量
        $\langle$告警+日志, 修复操作$\rangle$ 对——恰好符合 MOLE1
        所需的 $\langle$输入, 输出$\rangle$ 格式——无需额外标注工作量。
  \item \textbf{客观二值验证。}
        "所提修复方案是否解决了故障？"这一真值信号已存于历史记录，
        可通过故障回放实现完全自动化的离线评估，无需线上环境。
  \item \textbf{可量化的延迟优势。}
        故障响应对延迟极为敏感。本地部署 SLM 响应时延 $<$100\,ms；
        云端 LLM API 调用耗时 500\,ms--2\,s，峰值故障风暴时更会超时。
        10--20$\times$ 的延迟差距是 MOLE1 直接解决的具体工业痛点。
  \item \textbf{战略双覆盖。}
        本方案同时直接面向课题一（LLM–SLM 协同机制）与课题十（故障应急响应 AIOps），
        展示了一套方法论能同时解决两大工业痛点的可能性。
\end{enumerate}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{研究目标}

本项目设定五项紧密关联的研究目标：

\begin{enumerate}[leftmargin=*, label=\textbf{O\arabic*.}, itemsep=3pt]
  \item \textbf{MOLE1 流水线设计。}
        设计并实现完整的冷启动流水线，涵盖自动化 CoT 生成与 SLM 蒸馏，
        包括答案验证过滤机制。
  \item \textbf{AIOps 验证。}
        在真实或接近真实的 AIOps 数据集上验证流水线，证明 MOLE1 生产的领域专家 SLM
        在故障诊断与修复基准上能达到或超越通用 LLM 的性能。
  \item \textbf{MCP 封装。}
        将每个训练好的 SLM 打包为标准化的 MCP（Model Context Protocol）工具端点，
        实现与云端 LLM Planner 层的即插即用集成。
  \item \textbf{专业化机制分析。}
        提供严谨的实证与理论分析，阐释领域专注型 SLM 为何能在目标任务上超越通用 LLM
        （即"专业化优于泛化"现象）。
  \item \textbf{跨域迁移研究。}
        通过将相同流水线应用于至少一个 AIOps 以外的领域（如代码审查分类或
        客服工单分流），验证 MOLE1 的领域通用性，证明该方法的价值不局限于特定场景。
\end{enumerate}

\noindent\textbf{量化目标：}
MOLE1 生产的 SLM 应在 AIOps 故障诊断基准上达到 GPT-4 级 LLM 修复操作准确率的 5\% 以内，
将单次推理延迟降低 $>$10$\times$（从 $\sim$1\,s 降至 $<$100\,ms），并将单查询 token 成本降至零（本地推理）。
学术产出：在 CCF-A 或同等顶级会议（如 NeurIPS、ICML、ICSE、FSE）发表论文 2 篇；
就冷启动 CoT 合成与 MCP 端点封装申请发明专利 1--2 项；系统在阿里巴巴 3 个以上 AIOps 业务场景中完成部署。

% ──────────────────────────────────────────────────────────────────────────────
\subsection{研究方法}

\subsubsection*{核心思路：MOLE1——答案验证 CoT 合成}

本文提出 \textbf{MOLE1}（\textbf{M}odel-\textbf{O}utput-\textbf{L}abel-\textbf{E}licited
CoT Synthesis，第一迭代），一套领域无关的流水线，能\emph{无需任何人工标注或具备领域知识
的教师模型}，将原始 $\langle$输入, 输出$\rangle$ 对转化为 CoT 标注训练数据。
其核心创新在于\emph{以已知答案作为自动验证器}：

{\small
\begin{align*}
\underbrace{\langle q,\, a \rangle}_{\text{冷启动数据}}
&\;\xrightarrow{\;\text{LLM},\;k\text{ 采样}\;}\;
\text{候选 CoT}
\;\xrightarrow{\;\text{答案过滤}\;}\;
\underbrace{\langle q,\, \text{CoT}^*,\, a \rangle}_{\text{验证三元组}}
\;\xrightarrow{\;\text{SFT+LoRA}\;}\;
\underbrace{\text{专家 SLM}}_{\text{MCP 端点}}
\end{align*}}
\noindent 其中 $q$ = 查询，$a$ = 标准答案，CoT$^*$ = 经验证的推理链。

中型模型（如 14B）通过温度采样（$\tau{=}0.8$）为每条查询生成 $k$ 条候选推理链。
仅保留最终结论与标准答案一致的推理链——这一通过/淘汰准则\emph{无需任何领域知识}。
经验证的三元组随后对一个紧凑型 SLM（简单子任务选用 1.8B，复杂子任务选用 4B）
进行专域微调。

\textbf{领域通用性。}
尽管我们以 AIOps 作为验证场景，该流水线\emph{完全领域无关}：仅要求 (i) 存在历史
$\langle$输入, 输出$\rangle$ 对，且 (ii) 答案正确性可自动验证。该条件在工业界广泛成立——
客服工单分流、代码审查分类、供应链异常检测——使 MOLE1 成为 LLM–SLM 协同的通用冷启动引擎，
远不止于 AIOps 场景。选择 AIOps 领域作为主要验证场景，是因为它唯一性地同时满足上述三项
结构性优势，并提供了清晰、可复现的实验协议。

\subsubsection*{四阶段流水线}

图~\ref{fig:pipeline} 展示了完整的 MOLE1 流水线。

\begin{figure}[H]
\centering
\input{fig_pipeline.tikz}
\caption{MOLE1 四阶段流水线：从冷启动数据自动化生产 SLM。}
\label{fig:pipeline}
\end{figure}

\begin{description}[leftmargin=0pt, itemsep=4pt]
  \item[第一阶段——数据采集。]
        从运维日志中收集 $\langle$告警+日志, 修复操作$\rangle$ 对。
        无需人工标注；数据来源于现有故障管理系统。
  \item[第二阶段——CoT 候选生成。]
        针对每条查询，14B 生成 $k$ 条候选推理链
        （温度 $\tau{=}0.8$，top-$p{=}0.95$）。该步骤对生成模型\emph{无}领域知识要求。
  \item[第三阶段——答案验证过滤。]
        对每条候选 CoT 进行评估：最终结论是否与记录的修复操作一致？
        匹配的 CoT 保留，不匹配的丢弃。
        保留率随任务复杂度而异；STaR \citep{star} 中的类似过滤在可比设置下保留率为 30--70\%。
        需注意数据质量是前提：第一阶段中，涉及操作员失误或修复标签有误的记录
        将通过轻量异常过滤器排除。
  \item[第四阶段——SLM 微调与 MCP 打包。]
        经验证的三元组通过带 LoRA 的监督微调（SFT）训练紧凑型 SLM。
        训练好的模型被封装为 MCP 合规工具端点，并注册到 Planner 层。
\end{description}

\subsubsection*{实验设计与评估方案}

\textbf{数据集。}
采用以下两类数据结合：(a) 来自阿里巴巴运维日志的内部 AIOps 数据集（须遵循与阿里巴巴
SRE 团队共同商定的数据共享协议）；(b) 公开可用的 AIOps-2022 基准 \citep{aiops}（保障可复现性）。
实验覆盖三个 AIOps 子场景：数据库层故障、网络分区事件和容器资源耗尽。
针对公开代码审查数据集的跨域迁移实验将验证 MOLE1 的领域通用性（目标 O5）。

\textbf{对比基线。}
\begin{itemize}[itemsep=1pt]
  \item \textbf{仅 LLM}：通过 API 调用 72B，无本地 SLM。
  \item \textbf{SLM（无 CoT）}：SLM 仅在 $\langle$输入, 输出$\rangle$ 对上微调，无推理链。
  \item \textbf{SLM + 教师 CoT}：SLM 在由 72B 作为教师生成的 CoT 上微调
        （需要强能力教师，为``oracle''上界）。
  \item \textbf{MOLE1（本方案）}：SLM 在 14B 生成的答案验证 CoT 上微调。
\end{itemize}

\textbf{评估指标。}
修复操作准确率（top-1 命中率）、推理延迟（p50/p99）及单查询计算成本（FLOPs / API token）。
统计显著性通过自助重采样（$n{=}1000$）评估。

\subsubsection*{系统架构：Planner + MCP 工具层}

图~\ref{fig:arch} 展示了整体部署架构。

\begin{figure}[H]
\centering
\input{fig_arch.tikz}
\caption{系统架构：云端 LLM 作为 Planner，统筹调度由 MOLE1 流水线生产的本地 MCP 端点 SLM。}
\label{fig:arch}
\end{figure}

云端 LLM（72B 或同等模型）充当 \textbf{Planner}：它将输入的故障事件分解为子任务，
确定执行顺序，并调用相应的 MCP 工具端点。
每个 MCP 工具封装一个专为单一 AIOps 子场景训练的 SLM
（如数据库慢查询诊断、网络分区检测、容器 OOM 分诊）。

Planner–Executor 分层设计体现了两种角色的非对称性：
\begin{itemize}[itemsep=2pt]
  \item Planner 需要\emph{全局语义理解}，以分解新颖的多步故障事件——
        这种能力依赖大模型的广度。
  \item 每个 Executor 需要\emph{深度、窄域专业知识}，专注单一子领域——
        这种能力受益于在领域专属 CoT 数据上的专注训练，
        \emph{不}受益于大模型多任务训练所强加的泛化约束。
\end{itemize}

该设计同时带来延迟优势：仅 Planner 调用经过云端 API（$\sim$500\,ms）；
所有 Executor SLM 推理均在本地运行（$<$100\,ms），因此即便在高告警量下，
端到端系统也能在秒级内响应。

\subsubsection*{专家 SLM 为何能超越通用 LLM}

"专业化优于泛化"现象在蒸馏文献 \citep{dss,ddk} 中已有充分支撑，源于两点观察：
(i) 通用 LLM 的参数在数千个任务间共享，稀释了单一领域的表示容量 \citep{slmsurvey}；
(ii) 专注于领域 CoT 数据微调的 SLM 占据参数空间中更低维的子空间，
与目标领域推理模式高度对齐，在参数量远少的情况下实现更高精度。
我们将通过实证证据（消融研究，变量为模型规模与训练数据领域纯度）和容量分配分析，
对上述直觉进行形式化论证。

\textbf{为何不直接用 14B 中型模型作为 Executor？}
有人会问：既然 14B 已具备 CoT 能力，且可在单张 A100 GPU 上本地运行，
为何还要进一步蒸馏到 1.8B/4B 模型？
原因有三：(1) \emph{准确率}：专注于领域验证 CoT 训练的 1.8B/4B SLM 在目标子任务上
优于 14B 通用模型——该尺度下领域纯度效应主导了规模效应 \citep{ddk}；
(2) \emph{部署规模}：在拥有数百台主机的大型平台中，1.8B/4B SLM 可与每个服务节点
共同部署，内存开销可忽略，实现零延迟本地推理，无需 GPU 调度；
(3) \emph{成本}：1.8B 推理 FLOPs 约为 14B 的 $\approx$1/8（4B 约为 1/3.5），
并支持动态选择——简单子任务使用 1.8B Executor，复杂子任务调用 4B Executor——
对于高频故障监控（每分钟数千事件）至关重要。

\subsubsection*{技术优势}

将 MOLE1 与现有 LLM--SLM 协同及 SLM 蒸馏方法对比：

\begin{enumerate}[leftmargin=*, itemsep=4pt]
  \item \textbf{创新性。}
        现有 LLM--SLM 协同框架（FrugalGPT、RouteLLM）假设专家 SLM 已存在，
        完全聚焦于路由策略。现有 CoT 蒸馏方法（STaR、Distilling Step-by-Step）
        需要人工标注 CoT 或具备领域知识的教师模型。
        MOLE1 首次填补了冷启动空白：\emph{无需人工标注、无需领域感知教师}，
        仅以历史记录中已有的标准答案作为验证器，实现专家 SLM 的自动化生产。
        这填补了两条研究脉络中的关键盲点。
  \item \textbf{可部署性。}
        流水线设计在操作上高度透明：收集历史记录、运行温度采样 CoT 生成、
        按答案匹配过滤、LoRA 微调、注册为 MCP 端点。
        任何阿里巴巴 SRE 团队均可在数天内将流水线应用于新 AIOps 子场景，
        无需机器学习专业知识或标注预算。
        MCP 合规封装确保与现有 Planner 基础设施的即插即用集成。
  \item \textbf{有效性。}
        领域专属 CoT 训练产出的 SLM，在窄领域任务上能与 GPT-4 级 LLM 持平甚至超越，
        尽管参数量相差数量级——这是蒸馏文献 \citep{dss,ddk} 中的成熟结论，
        MOLE1 将其推广至冷启动答案验证场景。
        "专业化优于泛化"现象的实证与理论分析将作为项目成果的一部分输出。
  \item \textbf{高效性。}
        MOLE1 标注开销为零，所需算力仅为训练或调用通用 LLM 的一小部分。
        推理阶段收益立竿见影：本地 SLM 执行（$<$100\,ms）消除了高频 AIOps
        监控任务的云端 API 成本与延迟，流水线开销远小于其替代的 LLM API 费用。
  \item \textbf{通用性。}
        MOLE1 仅需具有可自动验证答案的历史 $\langle$输入, 输出$\rangle$ 对——
        这一条件在工业界广泛成立（客服工单分流、代码审查分类、供应链异常检测）。
        设计完成后，可无需领域专属再工程即扩展至阿里巴巴多条业务线，
        成为广泛可复用的冷启动资产，远超本项目所演示的 AIOps 场景。
\end{enumerate}

  \item \textbf{设计文档。}
        提交 1 份总体设计文档（系统架构与流水线规格）及 1 份详细设计文档
        （算法、评估方案与部署指南）。

        在 CCF-A 或同等顶级会议（如 NeurIPS、ICML、ACL、ICSE、FSE）

  \item \textbf{专利目标。}
        就冷启动 CoT 合成与 MCP 端点封装提交发明专利申请 1--2 项（国内或国际）。

        与现有 AIOps 方案对比：MOLE1 生产的 SLM 修复操作准确率在 GPT-4 级 LLM

        单查询 token 成本降至零（本地推理）。
  \item \textbf{业务目标。}
        系统在阿里巴巴 3 个以上 AIOps 业务场景完成部署上线，
        并有文档记录相对仅 LLM 基线的性能提升。

        项目结束时提交 1 份最终验收测试报告。

        联合培养参与本项目的研究生 2--3 名；在项目期间提供赴阿里巴巴
        SRE 团队实习机会（预计第 4--9 月，以阿里巴巴排期为准）。
