\section{Introduction}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{Title of Project}

\textbf{Automated Production of MCP-Endpoint Expert Models via Large-Model Cold-Start
Distillation for AIOps Fault Emergency Response}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{Background and Significance}

\textbf{Industrial motivation.}
Large-scale internet platforms such as Alibaba operate complex, heterogeneous
infrastructure that must maintain near-zero downtime. Fault-emergency response (AIOps)
is therefore a critical operational capability, yet it remains highly labor-intensive:
on-call engineers must diagnose alerts, correlate logs across dozens of subsystems,
and prescribe remediation actions under severe time pressure.
The Qwen3 model family \citep{qwen3} and the broader maturation of Large Language Models
(LLMs) open the door to automating this workflow.
However, a single cloud-hosted LLM incurs API latency of 500\,ms–2\,s per call—and
can time out entirely during high-volume fault storms—whereas locally deployed small
language models (SLMs) respond in $<$100\,ms, making them indispensable for latency-
sensitive AIOps execution.

\textbf{The hidden assumption in LLM–SLM collaboration research.}
Topic 1 of this scheme asks how large and small models can collaborate effectively.
Existing collaboration frameworks (e.g., FrugalGPT, RouteLLM) implicitly assume that
\emph{specialist small models already exist}: their contribution is an efficient
routing or scheduling policy that decides which model handles a given query.
They do not answer the more fundamental question: \emph{where do specialist small
models come from?}

\textbf{Reasoning-chain training as the established path to capable SLMs.}
A separate line of work—STaR \citep{star}, Distilling Step-by-Step \citep{dss}, and
related methods \citep{minillm,ddk,onpolicy}—has convincingly demonstrated that
equipping an SLM with \emph{chain-of-thought (CoT) reasoning} during training, rather
than merely memorising input–output pairs, dramatically improves its performance on
domain-specific tasks.
The bottleneck is therefore data: obtaining high-quality CoT annotations requires
either expensive human annotators or a large teacher model that can generate rationales
on demand.
Neither resource is available at \emph{cold-start}, i.e., when a team launches a new
business domain with no labelled data and no pre-existing expert model.

\textbf{The research gap.}
As illustrated in Figure~\ref{fig:gap}, existing SLM-production methods all require
at least one of: (a) human-labelled CoT, (b) a teacher model with prior knowledge of
the domain, or (c) an already-performing student model to bootstrap from.
In cold-start settings—precisely the condition that governs new AIOps sub-scenarios—
none of these prerequisites hold.
This gap motivates our core proposal: an automated pipeline that generates
CoT-annotated training data \emph{from raw} $\langle$\emph{input, output}$\rangle$
\emph{pairs alone}, requiring no human intervention and no domain-aware teacher.

\begin{figure}[H]
\centering
\input{fig_gap.tikz}
\caption{Existing SLM-production methods fail at cold-start (left); MOLE1 succeeds
by using the observed answer as an automatic verifier (right).}
\label{fig:gap}
\end{figure}

\textbf{Why AIOps is the ideal proving ground.}
Three structural properties make AIOps the best domain in which to validate the proposed
approach.
\begin{enumerate}[leftmargin=*, itemsep=2pt]
  \item \textbf{Data abundance without labelling cost.}
        Every large-scale platform accumulates rich
        $\langle$alert+log, remediation-action$\rangle$ pairs in its operational
        databases—precisely the $\langle$input, output$\rangle$ format MOLE1
        requires—with zero additional annotation effort.
  \item \textbf{Objective, binary verification.}
        ``Did the proposed remediation resolve the fault?'' is a ground-truth signal
        available in historical records, enabling fully automated offline evaluation
        via incident replay—no live environment needed.
  \item \textbf{Quantifiable latency advantage.}
        Fault response is acutely latency-sensitive. A locally deployed SLM responds
        in $<$100\,ms; a cloud LLM API call costs 500\,ms--2\,s and times out under
        peak-load fault storms. This 10--20$\times$ gap is a concrete, measurable
        industrial pain point that MOLE1 directly addresses.
  \item \textbf{Strategic dual coverage.}
        The proposed method directly addresses both Topic 1 (LLM–SLM collaboration
        mechanism) and Topic 10 (fault emergency response AIOps), demonstrating that
        a principled methodology can simultaneously solve two industrial pain points.
\end{enumerate}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{Objective of Project}

This project pursues four tightly coupled objectives:

\begin{enumerate}[leftmargin=*, label=\textbf{O\arabic*.}, itemsep=3pt]
  \item \textbf{MOLE1 pipeline design.}
        Design and implement the full cold-start pipeline for automated CoT generation
        and SLM distillation, including the answer-verification filtering mechanism.
  \item \textbf{AIOps validation.}
        Validate the pipeline on real or realistic AIOps datasets, demonstrating that
        domain-specialist SLMs produced by MOLE1 match or outperform general LLMs on
        fault diagnosis and remediation benchmarks.
  \item \textbf{MCP encapsulation.}
        Package each trained SLM as a standardised
        Model-Context-Protocol (MCP) tool endpoint, enabling plug-and-play integration
        with a cloud LLM Planner layer.
  \item \textbf{Specialisation mechanism analysis.}
        Provide rigorous empirical and theoretical analysis explaining \emph{why} a
        domain-focused SLM can outperform a general LLM on targeted tasks (the
        ``specialisation beats generalisation'' phenomenon).
\end{enumerate}

\subsection{Development Method}

\subsubsection*{Core Insight: MOLE1—Answer-Verified CoT Synthesis}

We propose \textbf{MOLE1} (\textbf{M}odel-\textbf{O}utput-\textbf{L}abel-\textbf{E}licited
CoT Synthesis, iteration 1), a domain-agnostic pipeline that converts raw
$\langle$input, output$\rangle$ pairs into CoT-annotated training data \emph{without
any human annotation or domain-aware teacher model}.
The key innovation is using the \emph{known answer as an automatic verifier}:

\[
\underbrace{\langle \text{query},\, \text{answer} \rangle}_{\text{available cold-start}}
\;\xrightarrow{\text{medium LLM, }k\text{ samples}}\;
\text{candidate CoTs}
\;\xrightarrow{\text{answer-consistency filter}}\;
\underbrace{\langle \text{query},\, \text{CoT}^*, \text{answer} \rangle}_{\text{verified triples}}
\;\xrightarrow{\text{SLM fine-tune}}\;
\text{specialist SLM}
\]

A medium-sized model (e.g., Qwen3-7B) generates $k$ candidate reasoning chains per
query via temperature sampling ($\tau{=}0.8$). Only chains whose final conclusion
matches the ground-truth answer are retained—a pass/fail criterion that requires
\emph{no domain knowledge}. The verified triples fine-tune a compact SLM (e.g.,
Qwen3-1.5B/3B) exclusively on the target domain.

\textbf{Domain generality.}
Although we validate MOLE1 on AIOps, the pipeline is \emph{entirely
domain-agnostic}: it requires only that (i) historical
$\langle$input, output$\rangle$ pairs exist and (ii) answer correctness can be
verified automatically. This condition holds across a wide range of industrial
scenarios—customer-service triage, code-review classification, supply-chain
anomaly detection—making MOLE1 a general-purpose cold-start engine for
LLM–SLM collaboration, well beyond its AIOps instantiation.
The AIOps domain is chosen as the primary validation testbed because it uniquely
satisfies all three structural advantages described above and provides a clear,
reproducible experimental protocol.

\subsubsection*{Four-Stage Pipeline}

Figure~\ref{fig:pipeline} shows the complete MOLE1 pipeline.

\begin{figure}[H]
\centering
\input{fig_pipeline.tikz}
\caption{MOLE1 four-stage pipeline for automated SLM production from cold-start data.}
\label{fig:pipeline}
\end{figure}

\begin{description}[leftmargin=0pt, itemsep=4pt]
  \item[Stage 1 – Data Harvesting.]
        Collect $\langle$alert+log, remediation$\rangle$ pairs from operational logs.
        No manual labelling; records come from existing incident-management systems.
  \item[Stage 2 – CoT Candidate Generation.]
        For each query, Qwen3-7B generates $k$ candidate reasoning chains
        (temperature $\tau{=}0.8$, top-$p{=}0.95$). This step requires no domain
        knowledge on the part of the generating model.
  \item[Stage 3 – Answer-Verification Filtering.]
        Each candidate CoT is evaluated: does the final conclusion match the recorded
        remediation action? Matched CoTs are retained; mismatched ones are discarded.
        Acceptance rate varies with task complexity; analogous filtering in STaR
        \citep{star} yields 30--70\% retention under comparable settings.
        We note that data quality is a prerequisite: records involving operator error
        or mislabelled remediation are excluded via a lightweight anomaly filter in
        Stage 1.
  \item[Stage 4 – SLM Fine-Tuning and MCP Packaging.]
        The verified triples train a compact SLM via supervised fine-tuning (SFT) with
        LoRA. The trained model is wrapped as an MCP-compliant tool endpoint and
        registered with the Planner layer.
\end{description}

\subsubsection*{Experimental Design and Evaluation Protocol}

\textbf{Dataset.}
We use a combination of (a) a proprietary AIOps dataset from Alibaba operational
logs (subject to data-sharing agreement) and (b) the publicly available
AIOps-2022 benchmark \citep{aiops} for reproducibility.
Experiments cover three AIOps sub-scenarios: database-layer faults, network
partition events, and container resource exhaustion.

\textbf{Baselines.}
\begin{itemize}[itemsep=1pt]
  \item \textbf{LLM-only}: Qwen3-72B queried via API, no local SLM.
  \item \textbf{SLM w/o CoT}: SLM fine-tuned on $\langle$input, output$\rangle$ pairs
        only (no reasoning chains).
  \item \textbf{SLM + teacher CoT}: SLM fine-tuned with CoT generated by Qwen3-72B
        acting as teacher (the ``oracle'' upper bound requiring a capable teacher).
  \item \textbf{MOLE1 (ours)}: SLM fine-tuned on answer-verified CoT from Qwen3-7B.
\end{itemize}

\textbf{Metrics.}
Remediation-action accuracy (top-1 match rate), inference latency (p50/p99),
and compute cost per query (FLOPs / API tokens).
Statistical significance assessed via bootstrap resampling ($n{=}1000$).

\subsubsection*{System Architecture: Planner + MCP Tool Layer}

Figure~\ref{fig:arch} depicts the overall deployment architecture.

\begin{figure}[H]
\centering
\input{fig_arch.tikz}
\caption{System architecture: cloud LLM as Planner orchestrates locally deployed
MCP-endpoint SLMs produced by the MOLE1 pipeline.}
\label{fig:arch}
\end{figure}

The cloud-hosted LLM (Qwen3-72B or equivalent) acts as a \textbf{Planner}: it
decomposes incoming fault events into sub-tasks, determines execution order, and
invokes the appropriate MCP tool endpoints.
Each MCP tool wraps an SLM that has been trained exclusively for one AIOps sub-scenario
(e.g., database slowdown diagnosis, network partition detection, container OOM triage).

The Planner–Executor split reflects the asymmetric nature of the two roles:
\begin{itemize}[itemsep=2pt]
  \item The Planner requires \emph{global semantic understanding} to decompose novel,
        multi-step fault events—a capability that demands the breadth of a large model.
  \item Each Executor requires \emph{deep, narrow expertise} in a single sub-domain—
        a capability that benefits from focused training on domain-specific CoT data,
        and which does \emph{not} benefit from the generalisation enforced by
        multi-task training on a large model.
\end{itemize}

This design also delivers the latency advantage: only the Planner call traverses the
cloud API ($\sim$500\,ms); all Executor SLM inferences run locally ($<$100\,ms), so
the end-to-end system responds in seconds even under high alert volume.

\subsubsection*{Why a Specialist SLM Can Outperform a General LLM}

The ``specialisation beats generalisation'' phenomenon is well-supported by the
distillation literature \citep{dss,ddk} and follows from two observations:
(i) a general LLM's parameters are shared across thousands of tasks, diluting
per-domain capacity; (ii) an SLM trained exclusively on domain-specific CoT data
develops tightly aligned internal representations for that domain, achieving higher
precision despite far fewer parameters.
We will provide both empirical evidence (ablation studies) and a capacity-allocation
analysis to formalise this intuition.
