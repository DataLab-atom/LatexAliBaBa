\section{项目介绍}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{项目标题}

\textbf{基于大模型冷启动蒸馏的 MCP 端点专家模型自动化生产——面向 AIOps 故障应急响应}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{背景与意义}

\textbf{产业背景。}
阿里巴巴等大规模互联网平台须保障近乎零停机时间。
以电商大促场景为例，单次高峰期可触发数万条告警，人工 triage 每条平均耗时 5--15 分钟，%
\footnote{基于阿里巴巴内部大促运维报告估算；类似规模见 \citet{aiops}。}
产生数百工程师-小时的应急成本，且高压情境下人工误判率显著上升。
Qwen3 系列模型 \citep{qwen3} 的成熟为该流程的自动化打开了大门，
然而云端 LLM API 单次调用延迟高达 500\,ms--2\,s，在高频故障风暴下极易超时；
而本地部署的小语言模型（SLM）响应时延 $<$100\,ms，对延迟敏感的 AIOps 执行层不可或缺。

\textbf{LLM--SLM 协同中被忽视的前提。}
课题一要求探索大小模型如何有效协同。
现有协同框架（如 FrugalGPT、RouteLLM）隐含假设\emph{专家小模型已经存在}，
贡献的是路由或调度策略，却未回答更根本的问题：\emph{专家小模型从何而来？}
现有 SLM 生产方法——无论是基于人工标注思维链（CoT）的监督微调 \citep{dss}、
基于教师模型输出分布对齐的知识蒸馏 \citep{minillm,ddk}，
还是令模型自我采样推理链再过滤的自举方法 \citep{star}——均依赖以下至少一项：
(a)~人工标注 CoT；(b)~具备领域知识的教师模型；(c)~已有非平凡解题能力的学生模型。
而在\emph{冷启动}场景——新 AIOps 子场景的常态——上述资源均不可得（图~\ref{fig:gap}）。

\begin{figure}[H]
\centering
\input{fig_gap.tikz}
\caption{现有 SLM 生产方法在冷启动场景下失效（左）；
本方案以历史记录中已有的标准答案作为自动验证器，绕过这一瓶颈（右）。}
\label{fig:gap}
\end{figure}

\textbf{核心问题。}
如何在\emph{零人工标注、无领域感知教师}的严格冷启动条件下，
仅从原始 $\langle$输入, 输出$\rangle$ 对自动生产具有推理能力的专家 SLM？
现有工作——规则引擎、ML 分类器或检索增强生成（RAG）——均未解决这一问题：
规则引擎无法泛化到新故障类型；ML 分类器缺乏可解释推理步骤；
RAG 每次仍须调用云端 LLM，未能消除延迟瓶颈。
本项目针对这一研究空白，拟设计并验证一套全自动的冷启动 SLM 生产流水线。

\textbf{为何 AIOps 是理想验证场景。}
三项结构性特点使 AIOps 成为验证本方案的最佳领域：
\begin{enumerate}[leftmargin=*, itemsep=2pt]
  \item \textbf{数据天然符合格式需求。}
        每个平台的运维数据库中均积累了大量
        $\langle$告警+日志, 修复操作$\rangle$ 对，恰好符合所需的
        $\langle$输入, 输出$\rangle$ 格式，无需额外标注工作量。
  \item \textbf{客观二值验证。}
        "所提修复方案是否解决了故障？"的真值信号已存于历史记录，
        可通过故障回放完全自动化评估，无需人工裁判或领域专家介入。
  \item \textbf{可量化的延迟价值。}
        本地 SLM 推理时延 $<$100\,ms，相较云端 API 的 500\,ms--2\,s，
        直接解决高频告警场景的工业痛点，量化指标清晰可复现。
\end{enumerate}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{研究目标}

本项目设定四项紧密关联的研究目标：

\begin{enumerate}[leftmargin=*, label=\textbf{O\arabic*.}, itemsep=3pt]
  \item \textbf{冷启动专家 SLM 生产框架。}
        设计并验证一套三循环迭代流水线，仅从原始 $\langle$输入, 输出$\rangle$ 对出发，
        经由答案验证 CoT 合成（Cycle 1）、监督推理训练（Cycle 2）、
        领域聚焦策略优化（Cycle 3）及自举（Bootstrap）回路，
        实现冷启动条件下专家 SLM 的全自动化生产。
  \item \textbf{AIOps 场景系统验证。}
        在阿里巴巴运维数据集及公开 AIOps 基准 \citep{aiops} 上验证所提流水线，
        证明本地专家 SLM 在故障诊断与修复准确率上能达到或接近通用大模型的水准。
  \item \textbf{专业化机制实证分析。}
        通过消融实验（变量为模型规模、训练数据领域纯度、CoT 过滤率）
        及表示空间分析（t-SNE 可视化、CKA 层间相似度），
        对"领域专属 CoT 训练使小模型在窄域任务上超越大模型"现象进行实证论证。
  \item \textbf{跨域适用性初探（探索性）。}
        在资源允许的条件下，将流水线应用于 AIOps 以外的公开数据集
        （如代码审查分类），初步验证本方法的领域无关性；
        若时间有限，此目标以消融实验形式完成并作为后续工作铺垫。
\end{enumerate}

\noindent\textbf{工程交付物（非研究目标）：}
将训练好的 SLM 封装为 MCP（Model Context Protocol）合规工具端点并集成至 Planner 层，
作为 O1 研究成果向生产环境迁移的必要工程出口，不构成独立研究贡献。

\noindent\textbf{量化目标：}
所产出 SLM 在 AIOps 故障诊断基准上达到 GPT-4 级 LLM 修复准确率的 5\% 以内，
单次推理延迟降低 $>$10$\times$（从 $\sim$1\,s 降至 $<$100\,ms），
单查询 token 成本降至零（本地推理）。
学术产出：投稿论文 2 篇至 CCF-A 或同等顶级会议（如 NeurIPS、ICML、ICSE、FSE），
争取于项目结题前至少 1 篇完成录用；
就冷启动 CoT 合成与 MCP 端点封装申请发明专利 1--2 项；
系统在 2 个以上 AIOps 子场景完成原型部署与性能验证。

% ──────────────────────────────────────────────────────────────────────────────
\subsection{研究方法}

\subsubsection*{核心思路：三循环迭代答案验证 CoT 合成}

本项目拟设计一套\textbf{三循环迭代 CoT 合成流水线}，领域无关，
\emph{无需任何人工标注或具备领域知识的教师模型}，
将原始 $\langle$输入, 输出$\rangle$ 对转化为具有推理能力的专家 SLM。
系统由三个相互依存的训练循环构成，并配有一个自举（Bootstrap）回路：

{\small
\begin{alignat}{2}
\text{Cycle 1:}&\quad
  \underbrace{\langle q,\, a \rangle}_{\text{冷启动数据}}
  &\;\xrightarrow{\;\text{最小教师},\;k\text{ 采样 + 答案过滤}\;}&\;
  \underbrace{\mathcal{D}^*}_{\text{验证三元组集}} \notag\\[4pt]
\text{Cycle 2:}&\quad
  \mathcal{D}^*
  &\;\xrightarrow{\;\text{SFT+LoRA}\;}&\;
  \underbrace{\pi_0}_{\text{初始 SLM}} \notag\\[4pt]
\text{Cycle 3:}&\quad
  \pi_0
  &\;\xrightarrow{\;\text{AIOps-GRPO}\;}&\;
  \underbrace{\pi^*}_{\text{专家 SLM}} \notag\\[4pt]
\text{Bootstrap:}&\quad
  \pi_t\bigl(\mathcal{D}_\emptyset\bigr)
  &\;\xrightarrow{\;\text{过滤}\;}\;
  \mathcal{D}^*_{\text{new}} \hookrightarrow \mathcal{D}^*
  &\;\xrightarrow{\;\text{触发下一轮 Cycle 1}\;}\;
  \pi_{t+1} \notag
\end{alignat}}

\noindent 其中 $q$ = 查询，$a$ = 标准答案，
$\mathcal{D}_\emptyset$ = 本轮过滤率为零的失败样本集，$\pi_t$ = 第 $t$ 轮训练所得 SLM。
这一设计的关键在于：以历史记录中\emph{已有}的标准答案作为唯一外部验证信号，
无需引入额外人工监督，即可自动筛选通向正确答案的推理链，从而将通用模型的生成能力
转化为经验证的领域特定训练数据。
在相邻科学计算领域，类似的迭代答案验证思路已展现出方法论上的可行性 \citep{logos}；
本项目的目标是系统探索该思路在 AIOps 冷启动场景下的适用条件、奖励信号设计与泛化能力。

\subsubsection*{三循环训练流程}

图~\ref{fig:pipeline} 展示了完整的三循环流水线。

\begin{figure}[H]
\centering
\input{fig_pipeline.tikz}
\caption{三循环迭代流水线：Cycle 1 产出验证数据集，Cycle 2 SFT 得到初始 SLM，
Cycle 3 策略优化强化推理质量，Bootstrap 回路驱动数据与模型协同进化。}
\label{fig:pipeline}
\end{figure}

\begin{description}[leftmargin=0pt, itemsep=6pt]

  \item[Cycle 1——数据采集与验证 CoT 合成。]
        从运维日志中收集 $\langle$告警+日志, 修复操作$\rangle$ 对，
        无需任何人工标注；数据来源于现有故障管理系统。

        针对每条查询，由\emph{任务可行的最小外部教师模型}
        （温度采样 $\tau{=}0.8$，top-$p{=}0.95$）生成 $k$ 条候选推理链。
        "任务可行的最小模型"指在目标子场景能以非平凡概率产出正确答案的最小可用模型；
        相比直接调用 72B 全量模型，可将生成成本降低约 $5{\times}$--$10{\times}$。
        选择\emph{外部}教师的根本原因在于：
        冷启动条件下目标 SLM 尚不具备解题能力，自举路径无法启动。

        对每条候选推理链，提取末尾预测操作序列 $\hat{a}_j = \mathrm{ans}(c_j)$
        与标准答案 $a = (o_1, \ldots, o_m)$ 进行集合等价匹配（规范化后）：
        \[
          \mathcal{D}^* = \bigl\{\,(q,\, c_j,\, a)\;\bigm|\;
            \mathrm{ans}(c_j) \equiv a,\; j = 1,\ldots,k\,\bigr\}
        \]
        操作序列的结构化性质（每个 $o_i$ 为 \texttt{restart\_service}、
        \texttt{scale\_pod} 等运维指令）保证匹配判断\emph{完全自动化}，无需领域知识。
        涉及操作员失误或标签模糊的记录通过轻量异常过滤器排除；
        保留率通常在 30--70\% 区间（$k{=}8$），足以为下一阶段提供高质量训练数据。

  \item[Cycle 2——有监督推理训练（SFT + LoRA）。]
        将 $\mathcal{D}^*$ 中的验证三元组 $\langle q, c^*, a \rangle$ 用于
        对紧凑型 SLM（简单子任务选用 1.5B，复杂子任务选用 4B）
        进行带 LoRA 的监督微调，最大化 CoT 与答案的联合自回归似然。
        损失函数对推理链过短（$|c| < \gamma$）的样本降权，
        防止模型旁路推理步骤直接输出答案。
        此阶段产出初始 SLM $\pi_0$，
        它已能遵循任务格式并输出合理推理链，但准确率尚待进一步强化。

  \item[Cycle 3——领域聚焦策略优化（AIOps-GRPO）。]
        仅靠 SFT 无法保证推理链导向正确操作——SFT 拟合的是采样分布，
        而非最优推理策略。
        对 $\pi_0$ 施加群体相对策略优化 \citep{onpolicy}：
        每条 prompt 生成 8 条 rollout，按奖励计算组内相对优势并更新策略参数。

        \textbf{AIOps 奖励设计。}
        AIOps 的结构化操作格式天然提供等效于领域验证器的信号：
        \begin{itemize}[itemsep=1pt]
          \item $\mathcal{R}_{\mathrm{format}}$：输出符合结构化操作列表格式；
          \item $\mathcal{R}_{\mathrm{CoT}}$：推理链长度超过最小阈值
                （激励实质性推理，而非直接输出答案）；
          \item $\mathcal{R}_{\mathrm{EM}}$：操作序列与标准答案精确匹配
                （复用 Cycle 1 的验证函数，无需额外实现）；
          \item $\mathcal{R}_{\mathrm{soft}}$：操作类型匹配但参数有差异时给予部分分；
          \item $\mathcal{R}_{\mathrm{len}}$（负项）：对过长推理链的软惩罚。
        \end{itemize}
        总奖励 $\mathcal{R}_{\mathrm{total}} = \sum_k \lambda_k \mathcal{R}_k$，
        $\mathcal{R}_{\mathrm{EM}}$ 与 $\mathcal{R}_{\mathrm{soft}}$ 的联合权重主导（约 0.5），
        KL 散度惩罚限制策略偏离初始点以保持训练稳定。

  \item[Bootstrap 回路——自举数据扩充。]
        当某子任务在 Cycle 1 过滤率为零（$\mathcal{D}^* = \emptyset$）时，
        训练数据完全为空，流水线陷入死锁。
        迭代架构天然处理这一情形：
        用\emph{当前已训练的 SLM}（$\pi_t$，而非外部教师）对失败样本重新采样，
        经答案验证过滤后写回 $\mathcal{D}^*$，触发下一轮迭代，
        实现数据集与模型能力的协同进化。
        在积累足够 Bootstrap 样本前，该子任务请求暂由 Planner LLM 直接处理，
        维持系统可用性而不产出空训练集。

\end{description}

\subsubsection*{实验设计与评估方案}

\textbf{数据集。}
采用两类数据结合：
(a)~来自阿里巴巴运维日志的内部 AIOps 数据集（已与 SRE 团队初步确认数据访问意向，
数据共享协议正在商定中），预估规模不低于 10,000 条故障事件，
以 8:1:1 比例切分训练/验证/测试集；
(b)~公开可用的 AIOps-2022 基准 \citep{aiops}（保障可复现性）。
实验覆盖三个子场景：数据库层故障、网络分区事件和容器资源耗尽。
CoT 候选生成阶段每条查询采样 $k{=}8$ 条推理链。

\textbf{对比基线。}
\begin{itemize}[itemsep=1pt]
  \item \textbf{仅 LLM}：通过 API 调用 72B，无本地 SLM。
  \item \textbf{SLM（无 CoT）}：SLM 仅在 $\langle$输入, 输出$\rangle$ 对上微调，无推理链。
  \item \textbf{SLM + 教师 CoT}：SLM 在由 72B 作为教师生成的 CoT 上微调（oracle 上界）。
  \item \textbf{本方案}：SLM 在答案验证 CoT 上微调，教师仅为任务可行的最小模型。
\end{itemize}

\textbf{评估指标。}
修复操作准确率（top-1 命中率）、推理延迟（p50/p99）及单查询计算成本（FLOPs / API token）。
统计显著性通过自助重采样（$n{=}1000$）评估。

\subsubsection*{系统架构：Planner + MCP 工具层}

图~\ref{fig:arch} 展示了整体部署架构。

\begin{figure}[H]
\centering
\input{fig_arch.tikz}
\caption{系统架构：云端 LLM 作为 Planner，统筹调度由三循环流水线离线生产的本地 MCP 端点 SLM。}
\label{fig:arch}
\end{figure}

云端 LLM（72B 或同等模型）充当 \textbf{Planner}：
它将输入的故障事件分解为子任务，确定执行顺序，并调用相应的 MCP 工具端点。
每个 MCP 工具封装一个专为单一 AIOps 子场景训练的 SLM
（如数据库慢查询诊断、网络分区检测、容器 OOM 分诊）。

Planner--Executor 分层设计体现了两种角色的非对称性：
Planner 需要\emph{全局语义理解}以分解新颖的多步故障事件——依赖大模型的广度；
每个 Executor 需要\emph{深度、窄域专业知识}——受益于领域专属 CoT 数据的专注训练，
而非大模型多任务训练所强加的泛化约束。
延迟优势同样显著：仅 Planner 调用云端 API（$\sim$500\,ms）；
所有 Executor 推理均在本地运行（$<$100\,ms），端到端系统可在秒级内响应高频告警。

领域专属 CoT 训练使 SLM 在窄域任务上与规模远大于其的通用大模型竞争，
这一现象在蒸馏文献中已有充分记录 \citep{dss,ddk}：
专注训练使参数占据更低维的领域特异子空间，在参数量远少的情况下实现更高的任务准确率。
此外，1.5B/4B SLM 可与服务节点共同部署，内存开销极低，
支持简单子任务使用 1.5B、复杂子任务使用 4B 的动态选择策略，
对高频故障监控（每分钟数千事件）至关重要。
本项目将通过消融实验与表示空间分析对上述机制进行系统验证。
