\section{项目简介}

\subsection{课题名称}

\textbf{面向 AIOps 故障应急响应的 MCP 端点专家模型自动化生产方法研究——基于大模型冷启动知识蒸馏}

\subsection{研究背景与意义}

\textbf{产业需求背景。}
阿里巴巴等大规模互联网平台运营着复杂异构的基础设施，对系统稳定性要求极高。
AIOps 故障应急响应是核心运维能力，却高度依赖人工：值班工程师需在严苛时限内完成告警诊断、跨子系统日志关联与处置决策。
Qwen3 系列模型\citep{qwen3}的成熟为该流程的自动化提供了现实条件。
然而，云端大语言模型（LLM）单次 API 调用延迟达 500\,ms--2\,s，故障风暴期间甚至直接超时；而本地部署的小语言模型（SLM）推理延迟可控制在 $<$100\,ms，在对延迟极度敏感的 AIOps 执行层具有不可替代的价值。

\textbf{LLM--SLM 协同研究中被忽视的前提假设。}
本计划课题一聚焦于大小模型的高效协同机制。现有协同框架（如 FrugalGPT、RouteLLM）
均隐含一个前提：\emph{领域专家小模型已经存在}，其贡献在于设计高效的路由或调度策略。
然而，这些工作均未回答更本质的问题：\emph{领域专家小模型从何而来？}

\textbf{链式推理训练是构建高能力 SLM 的成熟路径。}
STaR \citep{star}、Distilling Step-by-Step \citep{dss} 及相关工作 \citep{minillm,ddk,onpolicy}
已充分证明：在训练时为 SLM 提供\emph{思维链（CoT）推理}，而非仅记忆输入输出对，
能大幅提升其在领域任务上的表现。
然而，获取高质量 CoT 标注需要昂贵的人工标注或具备领域先验的大型教师模型——
而在\emph{冷启动}场景下（即全新业务子领域启动时），上述两类资源均不具备。

\textbf{研究空白。}
如图~\ref{fig:gap} 所示，现有 SLM 生产方法均至少依赖以下之一：
（a）人工标注的 CoT；（b）具有领域先验的教师模型；或（c）已有一定能力的学生模型作为引导。
在冷启动场景——恰恰是 AIOps 新子场景上线时的普遍条件——上述前提均不成立。
这一空白促成本课题的核心提案：一套\emph{仅从原始 $\langle$输入, 输出$\rangle$ 数据对出发}，
无需人工介入、无需领域感知教师模型的 CoT 标注与 SLM 蒸馏自动化流水线。

\begin{figure}[H]
\centering
\input{fig_gap.tikz}
\caption{现有 SLM 生产方法在冷启动场景下的失效（左）；MOLE1 利用已知答案作为自动验证器实现突破（右）。}
\label{fig:gap}
\end{figure}

\textbf{为何 AIOps 是最佳验证场景。}
AIOps 具备三项结构性优势，使其成为验证本方法最理想的领域。
\begin{enumerate}[leftmargin=*, itemsep=2pt]
  \item \textbf{数据丰富、无需标注。}
        大规模平台的运维数据库天然积累了海量
        $\langle$告警+日志, 处置操作$\rangle$ 数据对——正是 MOLE1 所需的
        $\langle$输入, 输出$\rangle$ 格式，无需额外标注成本。
  \item \textbf{客观、可自动化的验证信号。}
        "所提处置方案是否解决了故障？"是历史记录中直接可查的真值信号，
        支持通过故障回放进行全自动离线评测，无需动用线上环境。
  \item \textbf{延迟优势可量化。}
        故障响应对延迟极为敏感：本地 SLM 响应 $<$100\,ms，
        云端 LLM API 调用耗时 500\,ms--2\,s，峰值故障风暴期间甚至超时。
        这 10--20 倍的延迟差距是 MOLE1 直接解决的、可量化的产业痛点。
  \item \textbf{同时覆盖双课题。}
        本方法同时响应课题一（大小模型协同机制）与课题十（AIOps 故障应急响应），
        以一套方法论同时解决两个产业痛点。
\end{enumerate}

\subsection{研究目标}

本课题设立五项紧密耦合的研究目标：

\begin{enumerate}[leftmargin=*, label=\textbf{O\arabic*.}, itemsep=3pt]
  \item \textbf{MOLE1 流水线设计与实现。}
        设计并实现完整的冷启动 CoT 自动生成与 SLM 蒸馏流水线，
        包括答案验证过滤机制。
  \item \textbf{AIOps 场景验证。}
        在真实或近真实的 AIOps 数据集上验证流水线，
        证明 MOLE1 产出的领域专家 SLM 在故障诊断与处置基准上
        达到或超越通用 LLM 的表现。
  \item \textbf{MCP 封装与集成。}
        将每个训练好的 SLM 封装为符合 Model Context Protocol（MCP）标准的工具端点，
        实现与云端 LLM 规划层的即插即用集成。
  \item \textbf{专精化机制分析。}
        通过严格的实证与理论分析，解释\emph{为何}领域专注 SLM 能在目标任务上超越通用 LLM（"专精胜泛化"现象）。
  \item \textbf{跨领域迁移验证。}
        将 MOLE1 流水线应用于至少一个 AIOps 以外的领域（如代码审查分类或客服意图识别），
        验证方法的领域无关性，证明其通用价值。
\end{enumerate}

\subsection{研究方法}

\subsubsection*{核心思路：MOLE1——答案验证驱动的 CoT 合成}

我们提出 \textbf{MOLE1}（\textbf{M}odel-\textbf{O}utput-\textbf{L}abel-\textbf{E}licited
CoT Synthesis，第一版），一套\textbf{领域无关}的自动化流水线，
能从原始 $\langle$输入, 输出$\rangle$ 数据对出发合成 CoT 标注训练数据，
\emph{无需任何人工标注或领域感知教师模型}。
核心创新在于将\emph{已知答案用作自动验证器}：

{\small
\begin{align*}
\underbrace{\langle q,\, a \rangle}_{\text{冷启动数据}}
&\;\xrightarrow{\;\text{中等 LLM},\;k\text{ 次采样}\;}\;
\text{候选 CoT}
\;\xrightarrow{\;\text{答案一致性过滤}\;}\;
\underbrace{\langle q,\, \text{CoT}^*,\, a \rangle}_{\text{验证数据三元组}}
\;\xrightarrow{\;\text{SFT+LoRA}\;}\;
\underbrace{\text{专家 SLM}}_{\text{MCP 端点}}
\end{align*}}
\noindent 其中 $q$ = 查询，$a$ = 真值答案，CoT$^*$ = 通过验证的推理链。

中等规模模型（如 Qwen3-7B）对每条查询通过温度采样（$\tau{=}0.8$）生成 $k$ 条候选推理链，
仅保留最终结论与真值答案一致的推理链——这一筛选标准\emph{不依赖任何领域知识}，
在冷启动条件下完全可行。筛选得到的三元组用于对轻量 SLM（如 Qwen3-1.5B/3B）进行专项微调。

\textbf{领域无关性说明。}
尽管本课题以 AIOps 作为验证领域，MOLE1 流水线\emph{完全领域无关}：
仅要求（i）存在历史 $\langle$输入, 输出$\rangle$ 数据对，
且（ii）答案正确性可自动验证。
这一条件在广泛的工业场景中均成立——客服工单分类、代码审查、供应链异常检测等——
使 MOLE1 成为大小模型协同框架下通用的冷启动生产引擎，远超其 AIOps 实例化的范围。

\subsubsection*{四阶段流水线}

图~\ref{fig:pipeline} 展示了 MOLE1 的完整流水线。

\begin{figure}[H]
\centering
\input{fig_pipeline.tikz}
\caption{MOLE1 冷启动 SLM 自动生产四阶段流水线。}
\label{fig:pipeline}
\end{figure}

\begin{description}[leftmargin=0pt, itemsep=4pt]
  \item[阶段一——数据采集。]
        从运维系统采集 $\langle$告警+日志, 处置操作$\rangle$ 数据对，
        无需人工标注；数据来自现有的故障管理系统。
        同时运行轻量级异常过滤器，剔除操作员误操作或处置结果存疑的记录，
        确保数据质量。
  \item[阶段二——CoT 候选生成。]
        对每条查询，Qwen3-7B 以温度采样（$\tau{=}0.8$，top-$p{=}0.95$）
        生成 $k$ 条候选推理链。该步骤对生成模型无领域知识要求。
  \item[阶段三——答案验证过滤。]
        逐条评估候选 CoT 的最终结论是否与历史处置操作一致；
        一致者保留，不一致者丢弃。
        类似筛选在 STaR \citep{star} 中的保留率约为 30--70\%。
  \item[阶段四——SLM 微调与 MCP 封装。]
        以验证数据三元组通过 SFT + LoRA 对轻量 SLM 进行微调；
        训练完成后将模型封装为 MCP 标准工具端点，注册至规划层。
\end{description}

\subsubsection*{实验设计与评估方案}

\textbf{数据集。}
采用两类数据：（a）阿里巴巴运维日志内部数据集（数据使用须经双方协商的数据共享协议授权）；
（b）公开的 AIOps-2022 基准数据集\citep{aiops}（用于可复现性对比）。
实验覆盖三类 AIOps 子场景：数据库层故障、网络分区故障、容器资源耗尽。
另设跨领域迁移实验（目标 O5），在公开代码审查数据集上验证 MOLE1 的领域泛化能力。

\textbf{基准对比方案。}
\begin{itemize}[itemsep=1pt]
  \item \textbf{仅大模型（LLM-only）}：Qwen3-72B 云端 API，无本地 SLM。
  \item \textbf{无 CoT 的 SLM（SLM w/o CoT）}：仅在输入输出对上微调，无推理链。
  \item \textbf{教师 CoT 的 SLM（SLM + teacher CoT）}：以 Qwen3-72B 生成的 CoT 微调（需要大模型教师，为"上限"基准）。
  \item \textbf{MOLE1（本文方法）}：以 Qwen3-7B 生成的答案验证 CoT 微调。
\end{itemize}

\textbf{评估指标。}
处置动作准确率（Top-1 匹配率）、推理延迟（p50/p99）、单次查询计算成本（FLOPs / API token 数）。
统计显著性通过自助法重采样验证（$n{=}1000$）。

\subsubsection*{系统架构：规划层 + MCP 工具层}

图~\ref{fig:arch} 展示了整体部署架构。

\begin{figure}[H]
\centering
\input{fig_arch.tikz}
\caption{系统架构：云端 LLM 作为规划器，调度由 MOLE1 流水线生产的 MCP 端点专家 SLM。}
\label{fig:arch}
\end{figure}

云端 LLM（Qwen3-72B 或同等级模型）担任\textbf{规划器}：将入站故障事件分解为子任务，
确定执行顺序，并依次调用对应的 MCP 工具端点。
每个 MCP 工具封装一个专项 SLM，该 SLM 经 MOLE1 在单一 AIOps 子场景（如数据库慢查询诊断、
网络分区检测、容器 OOM 定位）上专项训练。

规划器--执行器分层设计反映了两类角色的非对称性：
\begin{itemize}[itemsep=2pt]
  \item 规划器需要\emph{全局语义理解}以分解复杂多步故障事件——这需要大模型的广度。
  \item 每个执行器需要\emph{单一子领域的深度专精}——这受益于领域专项 CoT 训练，
        而\emph{不受益于}大模型多任务训练所强制执行的泛化约束。
\end{itemize}

该设计同时带来延迟优势：仅规划器调用经过云端 API（$\sim$500\,ms）；
所有执行器 SLM 推理均在本地完成（$<$100\,ms），
即便在高频告警场景下系统整体响应仍维持在秒级以内。

\subsubsection*{为何领域专精 SLM 能超越通用 LLM}

"专精胜泛化"现象已得到蒸馏文献的充分支持\citep{dss,ddk}，
其原因在于：（i）通用 LLM 的参数被数千个任务共享，导致单领域的表示容量被稀释\citep{slmsurvey}；
（ii）在领域专项 CoT 数据上微调的 SLM 在参数空间中占据更低维的子空间，
与目标领域的推理模式高度对齐，以更少的参数实现更高的精度。
本课题将提供消融实验（变量：模型规模、训练数据领域纯度）与容量分配分析，对上述直觉进行形式化论证。

\textbf{为何不直接以 7B 模型作为执行器？}
或许有人会问：既然 Qwen3-7B 已具备 CoT 能力且可在本地单张 A10 卡上运行，
为何还要进一步蒸馏为 1.5B/3B 模型？
原因有三：（1）\emph{准确率}：在领域验证 CoT 上训练的 1.5B SLM 在目标子任务上超越 7B 通用模型——
领域纯度效应在此规模下主导尺寸效应\citep{ddk}；
（2）\emph{部署规模}：大型平台拥有数百台主机，1.5B SLM 可与各服务节点共部署，
内存开销极低，无需 GPU 调度即可实现零延迟本地推理；
（3）\emph{成本}：1.5B 推理的 FLOPs 约为 7B 的 $\frac{1}{5}$，
对于每分钟数千次故障告警的高频监控场景至关重要。

\noindent\textbf{预期技术指标：}
MOLE1 产出的 SLM 在 AIOps 故障诊断基准上准确率与 GPT-4 级 LLM 相差在 5\% 以内，
单次推理延迟降低 $>$10 倍（从 $\sim$1\,s 降至 $<$100\,ms），推理成本降至零（本地推理，无 API 费用）。

\noindent\textbf{对阿里巴巴的预期业务价值：}
一套可复用、领域无关的通用流水线，阿里 SRE 团队可在数天内将其应用于任意新 AIOps 子场景，
无需标注预算，无需预置专家模型。
