\section{项目介绍}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{项目标题}

\textbf{基于大模型冷启动蒸馏的 MCP 端点专家模型自动化生产——面向 AIOps 故障应急响应}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{背景与意义}

\textbf{产业背景。}
阿里巴巴等大规模互联网平台运维着复杂异构的基础设施，须保障近乎零停机时间。
以电商大促场景为例，单次高峰期可触发数万条告警，人工 triage 每条平均耗时 5--15 分钟，%
\footnote{基于阿里巴巴内部大促运维报告估算；类似规模见 \citet{aiops}。}
由此产生数百工程师-小时的应急处置成本，且人工误判率在高压情境下显著上升。
故障应急响应（AIOps）是关键运维能力，但目前仍高度依赖人工：值班工程师须在严峻时间压力下
诊断告警、跨数十子系统关联日志并给出修复方案。
Qwen3 系列模型 \citep{qwen3} 及 LLM（大语言模型）的整体成熟，为该流程的自动化打开了大门。
然而，单次云端 LLM API 调用延迟高达 500\,ms--2\,s，在大规模故障风暴下（每分钟数千事件）甚至会超时失败；
而本地部署的 SLM（小语言模型）响应时延 $<$100\,ms，对延迟敏感的 AIOps 执行层不可或缺。

\textbf{LLM–SLM 协同研究中的隐含前提。}
课题一要求探索大小模型如何有效协同。现有协同框架（如 FrugalGPT、RouteLLM）隐含假设
\emph{专家小模型已经存在}：其贡献在于高效的路由或调度策略，决定由哪个模型处理某次查询。
然而，这些工作并未回答更根本的问题：\emph{专家小模型从何而来？}

\textbf{推理链训练是构建高能力 SLM 的成熟路径。}
另一研究方向——STaR \citep{star}、Distilling Step-by-Step \citep{dss} 及相关方法
\citep{minillm,ddk,onpolicy}——已充分证明：在训练时为 SLM 引入\emph{思维链（CoT）推理}，
而非仅记忆输入输出对，能显著提升其在领域特定任务上的性能。
瓶颈在于数据：获取高质量 CoT 标注，需要昂贵的人工标注员或能按需生成推理链的大型教师模型。
在\emph{冷启动}场景下——即团队进入无标注数据、无现成专家模型的新业务领域时——上述资源均不可得。

\textbf{研究空白。}
如图~\ref{fig:gap} 所示，现有 SLM 生产方法均依赖以下至少一项：
(a) 人工标注 CoT；(b) 具备领域知识的教师模型；(c) 可用于引导的已有学生模型。
而在冷启动场景——恰恰是新 AIOps 子场景的常态——上述前提均不成立。
需要指出的是，现有 AIOps 方法（基于规则引擎、ML 分类器或检索增强生成）虽能处理
\emph{模式匹配}类故障，却不产出可本地部署的推理型 SLM：规则引擎无法泛化到新故障类型；
ML 分类器缺乏可解释的推理步骤；RAG 每次仍须调用云端 LLM，未能解决延迟问题。
这一空白激发了本项目的核心提案：一套\emph{仅从原始 $\langle$输入, 输出$\rangle$ 对}
自动生成 CoT 标注训练数据的流水线，无需任何人工介入，也无需具备领域知识的教师模型。

\textbf{现有问题与困难。}

\textbf{（1）冷启动场景下专家 SLM 生产"无自动化路径"。}
现有 LLM--SLM 协同框架将专家小模型视为已知前提：FrugalGPT、RouteLLM 等工作贡献
路由或调度策略，但不提供 Executor 本身的生产方法。
实践中，新 AIOps 子场景（如新引入的服务网格或新型微服务故障类型）初始时既无标注数据，
也无预训练领域模型。若缺乏从零自动化生产专家 SLM 的手段，整个协同技术栈将无 Executor 可路由。
如何构建一套完全自动化、无需标注的 SLM 生产流水线，仅从原始运维记录出发，是第一项核心挑战。

\textbf{（2）工业领域的答案验证 CoT 合成"仍未解决"。}
基于 CoT 的蒸馏方法——STaR \citep{star}、Distilling Step-by-Step \citep{dss}——已充分
证明推理链训练相比标准输入输出微调能产出更强的 SLM。然而，生成\emph{经验证}的 CoT 标注，
需要判断每条推理链是否导向正确答案，这又依赖人工评估者或领域专家教师模型。
冷启动时两者均不存在。如何仅凭历史记录中已有的标准答案，在无任何领域特定监督的情况下
自动验证候选推理链，且能泛化到工业 AIOps 各子场景，是第二项核心挑战。

本项目通过 MOLE1 流水线同时攻克上述两个问题，以阿里巴巴 AIOps 运营数据作为主要验证环境。

\label{fig:gap}

\textbf{为何 AIOps 是理想验证场景。}
三项结构性特点使 AIOps 成为验证本方案的最佳领域。
\begin{enumerate}[leftmargin=*, itemsep=2pt]
  \item \textbf{数据丰富，无标注成本。}
        每个大规模平台的运维数据库中均积累了大量
        $\langle$告警+日志, 修复操作$\rangle$ 对——恰好符合 MOLE1
        所需的 $\langle$输入, 输出$\rangle$ 格式——无需额外标注工作量。
  \item \textbf{客观二值验证。}
        "所提修复方案是否解决了故障？"这一真值信号已存于历史记录，
        可通过故障回放实现完全自动化的离线评估，无需线上环境。
  \item \textbf{可量化的延迟优势。}
        故障响应对延迟极为敏感。本地部署 SLM 响应时延 $<$100\,ms；
        云端 LLM API 调用耗时 500\,ms--2\,s，峰值故障风暴时更会超时。
        10--20$\times$ 的延迟差距是 MOLE1 直接解决的具体工业痛点。
  \item \textbf{战略双覆盖。}
        本方案同时直接面向课题一（LLM–SLM 协同机制）与课题十（故障应急响应 AIOps），
        展示了一套方法论能同时解决两大工业痛点的可能性。
\end{enumerate}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{研究目标}

本项目设定五项紧密关联的研究目标：

\begin{enumerate}[leftmargin=*, label=\textbf{O\arabic*.}, itemsep=3pt]
  \item \textbf{LLM--SLM 协同机制：冷启动 SLM 自动生产（MOLE1）。}
        设计并实现三循环迭代流水线，解决"专家 SLM 从何而来"这一
        LLM--SLM 协同框架中被忽视的前提问题：
        Cycle 1 答案验证 CoT 合成 + Cycle 2 SFT 监督训练 +
        Cycle 3 领域聚焦 GRPO + Bootstrap 自举回路，
        实现冷启动场景下专家 SLM 的全自动化生产。
  \item \textbf{AIOps 验证。}
        在真实或接近真实的 AIOps 数据集上验证流水线，证明 MOLE1 生产的领域专家 SLM
        在故障诊断与修复基准上能达到或超越通用 LLM 的性能。
  \item \textbf{专业化机制分析。}
        通过以下两类分析对"专业化优于泛化"现象进行严谨的实证论证：
        (i) \emph{消融实验}，系统控制变量为模型规模（1.5B / 4B / 14B）、
        训练数据领域纯度（纯领域 vs. 混合通用数据）及 CoT 过滤率（0\% / 30\% / 70\%）；
        (ii) \emph{表示空间分析}，采用 t-SNE 可视化对比专域 SLM 与通用 LLM 的激活聚类，
        并以层间激活相似度（CKA）量化各层特征表示的专业化程度。
  \item \textbf{跨域通用性验证（探索性）。}
        在项目资源允许的条件下，将流水线应用于一个 AIOps 以外的公开数据集
        （如代码审查分类），以初步验证 MOLE1 的领域无关性；
        若时间有限，此目标将以消融实验形式完成并作为后续工作的铺垫。
\end{enumerate}

\noindent\textbf{工程交付物（非研究目标）：}
将训练好的 SLM 封装为 MCP（Model Context Protocol）合规工具端点，注册至 Planner 层，
实现与现有云端 LLM 基础设施的即插即用集成。
MCP 封装本质上是标准化的工具调用接口工程，不构成独立研究贡献，
但作为将 MOLE1 研究成果部署至生产系统的必要工程出口而包含于项目交付物中。

\noindent\textbf{量化目标：}
MOLE1 生产的 SLM 应在 AIOps 故障诊断基准上达到 GPT-4 级 LLM 修复操作准确率的 5\% 以内，
将单次推理延迟降低 $>$10$\times$（从 $\sim$1\,s 降至 $<$100\,ms），并将单查询 token 成本降至零（本地推理）。
学术产出：投稿论文 2 篇至 CCF-A 或同等顶级会议（如 NeurIPS、ICML、ICSE、FSE），争取于项目结题前至少 1 篇完成录用；
就冷启动 CoT 合成与 MCP 端点封装申请发明专利 1--2 项；系统在 2 个以上 AIOps 子场景完成原型部署与性能验证。

% ──────────────────────────────────────────────────────────────────────────────
\subsection{研究方法}

\subsubsection*{核心思路：MOLE1——三循环迭代 CoT 合成}

本文提出 \textbf{MOLE1}（\textbf{M}odel-\textbf{O}utput-\textbf{L}abel-\textbf{E}licited
CoT Synthesis，第一迭代），一套领域无关的\emph{迭代}流水线，能\emph{无需任何人工标注或具备领域知识
的教师模型}，将原始 $\langle$输入, 输出$\rangle$ 对转化为具有推理能力的专家 SLM。
系统由三个相互依存的训练循环构成，并配有一个自举（Bootstrap）回路：

{\small
\begin{alignat}{2}
\text{Cycle 1:}&\quad
  \underbrace{\langle q,\, a \rangle}_{\text{冷启动数据}}
  &\;\xrightarrow{\;\text{14B},\;k\text{ 采样 + 答案过滤}\;}&\;
  \underbrace{\mathcal{D}^*}_{\text{验证三元组集}} \notag\\[4pt]
\text{Cycle 2:}&\quad
  \mathcal{D}^*
  &\;\xrightarrow{\;\text{SFT+LoRA}\;}&\;
  \underbrace{\pi_0}_{\text{初始 SLM（AIOps-0）}} \notag\\[4pt]
\text{Cycle 3:}&\quad
  \pi_0
  &\;\xrightarrow{\;\text{AIOps-GRPO}\;}&\;
  \underbrace{\pi^*}_{\text{专家 SLM}} \notag\\[4pt]
\text{Bootstrap:}&\quad
  \pi_t\bigl(\mathcal{D}_\emptyset\bigr)
  &\;\xrightarrow{\;\text{过滤}\;}\;
  \mathcal{D}^*_{\text{new}} \hookrightarrow \mathcal{D}^*
  &\;\xrightarrow{\;\text{触发下一轮 Cycle 1}\;}\;
  \pi_{t+1} \notag
\end{alignat}}

\noindent 其中 $q$ = 查询，$a$ = 标准答案，CoT$^*$ = 经验证的推理链，
$\mathcal{D}_\emptyset$ = 本轮过滤率为零的失败样本集，$\pi_t$ = 第 $t$ 轮训练所得 SLM。
Cycle 1 由任务可行的最小外部教师模型驱动；
Cycle 3 仅使用已有的答案匹配信号作为奖励，无需引入外部监督；
Bootstrap 回路则以当前 SLM 自身（而非教师模型）对失败样本重新采样，
所有过滤通过的新三元组写回 $\mathcal{D}^*$ 并触发下一迭代，
实现数据集与模型能力的协同进化。

\textbf{领域通用性。}
尽管我们以 AIOps 作为验证场景，该流水线\emph{完全领域无关}：仅要求 (i) 存在历史
$\langle$输入, 输出$\rangle$ 对，且 (ii) 答案正确性可自动验证。该条件在工业界广泛成立——
客服工单分流、代码审查分类、供应链异常检测——使 MOLE1 成为 LLM–SLM 协同的通用冷启动引擎，
远不止于 AIOps 场景。选择 AIOps 领域作为主要验证场景，是因为它唯一性地同时满足上述三项
结构性优势，并提供了清晰、可复现的实验协议。

\subsubsection*{三循环迭代训练系统}

图~\ref{fig:pipeline} 展示了完整的 MOLE1 流水线。

\begin{figure}[H]
\centering
\input{fig_pipeline.tikz}
\caption{MOLE1 三循环迭代流水线：Cycle 1 产出验证数据集，Cycle 2 SFT 得到初始 SLM，
Cycle 3 AIOps-GRPO 强化推理质量，Bootstrap 回路驱动协同进化。}
\label{fig:pipeline}
\end{figure}

\begin{description}[leftmargin=0pt, itemsep=6pt]

  \item[Cycle 1——数据采集与验证 CoT 合成。]
        从运维日志中收集 $\langle$告警+日志, 修复操作$\rangle$ 对。
        无需人工标注；数据来源于现有故障管理系统。

        针对每条查询，由\emph{任务可行的最小外部教师模型}（本项目以 14B 为参考）
        温度采样（$\tau{=}0.8$，top-$p{=}0.95$）生成 $k$ 条候选推理链。
        ``任务可行''意指该模型能以非平凡概率产出正确修复操作；
        根据子场景难度可动态选择 7B--32B 范围内满足条件的最小模型，
        相比直接调用 72B 全量 LLM 将生成成本降低约 $5{\times}$--$10{\times}$。
        选择\emph{外部}教师（而非令目标 SLM 自举）的根本原因在于：
        冷启动场景下 1.5B/4B 目标模型尚不具备解题能力——若它已具备，该场景就不再是冷启动问题。

        对每条候选 CoT，提取末尾预测操作序列 $\hat{a}_j = \mathrm{ans}(c_j)$
        与标准答案 $a = (o_1, \ldots, o_m)$ 进行集合等价匹配（$\equiv$，规范化后）：
        \[
          \mathcal{D}^* = \bigl\{\,(q,\, c_j,\, a)\;\bigm|\;
            \mathrm{ans}(c_j) \equiv a,\; j = 1,\ldots,k\,\bigr\}
        \]
        操作序列的结构化性质（每个 $o_i$ 为 \texttt{restart\_service}、
        \texttt{scale\_pod} 等结构化运维指令）保证匹配判断\emph{完全自动化}，
        无需任何领域知识。
        数据质量保障：涉及操作员失误或标签模糊的记录（多次重试后成功、部分修复）
        通过轻量异常过滤器排除，仅保留"单次操作后故障在窗口期 $T$ 内无复现"的高置信样本。
        保留率通常在 30--70\% 区间（$k{=}8$），已足够为 Cycle 2 提供高质量训练数据。

  \item[Cycle 2——有监督推理训练（SFT + LoRA）。]
        将 $\mathcal{D}^*$ 中的验证三元组 $\langle q, c^*, a \rangle$ 用于
        对紧凑型 SLM（简单子任务选用 1.5B，复杂子任务选用 4B）
        进行带 LoRA 的监督微调，最大化 CoT 与答案的联合自回归似然。
        损失函数对推理链过短（$|c| < \gamma$）的样本降权，
        防止模型旁路推理步骤直接输出答案。
        此阶段产出初始 SLM $\pi_0$（AIOps-0），
        它已能遵循任务格式并输出合理推理链，但化学（或运维）准确率尚未收敛。

  \item[Cycle 3——AIOps 聚焦策略优化（AIOps-GRPO）。]
        仅靠 SFT 无法保证 SLM 推理链导向正确操作——SFT 拟合的是教师采样分布，
        而非最优推理策略。Cycle 3 对 $\pi_0$ 施加群体相对策略优化（GRPO \citep{onpolicy}）：
        每条 prompt 生成 8 条 rollout，按奖励计算组内相对优势，更新策略参数。

        \textbf{AIOps 奖励设计。}
        相比分子科学需要 RDKit 化学验证器，AIOps 的结构化操作格式天然提供等效信号：
        \begin{itemize}[itemsep=1pt]
          \item $\mathcal{R}_{\mathrm{format}}$：输出符合结构化操作列表格式；
          \item $\mathcal{R}_{\mathrm{CoT}}$：推理链长度超过最小阈值
                （激励实质性推理，而非直接输出答案）；
          \item $\mathcal{R}_{\mathrm{EM}}$：操作序列与标准答案精确匹配
                （与 Cycle 1 使用同一验证函数，无需额外实现）；
          \item $\mathcal{R}_{\mathrm{soft}}$：操作类型匹配但参数有差异时给予部分分，
                提供稠密学习信号；
          \item $\mathcal{R}_{\mathrm{len}}$（负项）：对过长推理链的软惩罚，防止推理游荡。
        \end{itemize}
        总奖励 $\mathcal{R}_{\mathrm{total}} = \sum_k \lambda_k \mathcal{R}_k$，
        权重 $\boldsymbol{\lambda}$ 使 $\mathcal{R}_{\mathrm{EM}}$ 和 $\mathcal{R}_{\mathrm{soft}}$
        的联合权重主导（约 0.5），其余项各均分剩余权重。
        KL 散度惩罚限制策略偏离初始点，保持训练稳定。
        此阶段产出最终专家 SLM $\pi^*$，并封装为 MCP 工具端点注册至 Planner 层。

  \item[Bootstrap 回路——自举数据扩充。]
        当某子任务在 Cycle 1 过滤率为零（$\mathcal{D}^* = \emptyset$）时，
        STaR 的自蒸馏路径陷入死锁：无训练数据，模型无法改进。
        MOLE1 的迭代架构天然处理这一情形：
        用\emph{当前已训练的 SLM}（$\pi_t$，而非外部教师）对失败样本重新采样，
        通过答案验证过滤新的通过样本，将其写回 $\mathcal{D}^*$ 扩充数据集，
        触发下一轮 Cycle 1 $\to$ Cycle 2 $\to$ Cycle 3 迭代。
        这一自举回路使数据集与模型能力\emph{协同进化}：
        $\pi_t$ 比外部教师更了解当前训练分布，
        其采样对困难样本往往比初始教师具有更高通过率。
        在新子任务积累足够 Bootstrap 样本前，该子任务请求暂由 Planner LLM 直接处理，
        维持系统可用性而不产出空训练集。

\end{description}

\subsubsection*{实验设计与评估方案}

\textbf{数据集。}
采用以下两类数据结合：
(a) 来自阿里巴巴运维日志的内部 AIOps 数据集（已与阿里巴巴 SRE 团队初步确认
数据访问意向，数据共享协议正在商定中），预估规模不低于 10,000 条有标注修复操作的故障事件，
以 8:1:1 比例切分训练/验证/测试集；
(b) 公开可用的 AIOps-2022 基准 \citep{aiops}（保障可复现性）。
实验覆盖三个 AIOps 子场景：数据库层故障、网络分区事件和容器资源耗尽。
CoT 候选生成阶段每条查询采样 $k{=}8$ 条推理链，预期经答案验证过滤后
保留约 30--70\% 作为有效训练数据。
若项目进度允许，将对公开代码审查数据集开展探索性跨域实验（目标 O5）。

\textbf{对比基线。}
\begin{itemize}[itemsep=1pt]
  \item \textbf{仅 LLM}：通过 API 调用 72B，无本地 SLM。
  \item \textbf{SLM（无 CoT）}：SLM 仅在 $\langle$输入, 输出$\rangle$ 对上微调，无推理链。
  \item \textbf{SLM + 教师 CoT}：SLM 在由 72B 作为教师生成的 CoT 上微调
        （需要强能力教师，为``oracle''上界）。
  \item \textbf{MOLE1（本方案）}：SLM 在 14B 生成的答案验证 CoT 上微调。
\end{itemize}

\textbf{评估指标。}
修复操作准确率（top-1 命中率）、推理延迟（p50/p99）及单查询计算成本（FLOPs / API token）。
统计显著性通过自助重采样（$n{=}1000$）评估。

云端 LLM（72B 或同等模型）充当 \textbf{Planner}：它将输入的故障事件分解为子任务，
确定执行顺序，并调用相应的 MCP 工具端点。
每个 MCP 工具封装一个专为单一 AIOps 子场景训练的 SLM
（如数据库慢查询诊断、网络分区检测、容器 OOM 分诊）。

Planner–Executor 分层设计体现了两种角色的非对称性：
\begin{itemize}[itemsep=2pt]
  \item Planner 需要\emph{全局语义理解}，以分解新颖的多步故障事件——
        这种能力依赖大模型的广度。
  \item 每个 Executor 需要\emph{深度、窄域专业知识}，专注单一子领域——
        这种能力受益于在领域专属 CoT 数据上的专注训练，
        \emph{不}受益于大模型多任务训练所强加的泛化约束。
\end{itemize}

该设计同时带来延迟优势：仅 Planner 调用经过云端 API（$\sim$500\,ms）；
所有 Executor SLM 推理均在本地运行（$<$100\,ms），因此即便在高告警量下，
端到端系统也能在秒级内响应。

\subsubsection*{专家 SLM 为何能超越通用 LLM}

"专业化优于泛化"现象在蒸馏文献 \citep{dss,ddk} 中已有充分支撑，源于两点观察：
(i) 通用 LLM 的参数在数千个任务间共享，稀释了单一领域的表示容量 \citep{slmsurvey}；
(ii) 专注于领域 CoT 数据微调的 SLM 占据参数空间中更低维的子空间，
与目标领域推理模式高度对齐，在参数量远少的情况下实现更高精度。
我们将通过实证证据（消融研究，变量为模型规模与训练数据领域纯度）和容量分配分析，
对上述直觉进行形式化论证。

\textbf{为何不直接用 14B 中型模型作为 Executor？}
有人会问：既然 14B 已具备 CoT 能力，且可在单张 A100 GPU 上本地运行，
为何还要进一步蒸馏到 1.5B/4B 模型？
原因有三：(1) \emph{准确率}：专注于领域验证 CoT 训练的 1.5B/4B SLM 在目标子任务上
优于 14B 通用模型——该尺度下领域纯度效应主导了规模效应 \citep{ddk}；
(2) \emph{部署规模}：在拥有数百台主机的大型平台中，1.5B/4B SLM 可与每个服务节点
共同部署，内存开销可忽略，实现零延迟本地推理，无需 GPU 调度；
(3) \emph{成本}：1.5B 推理 FLOPs 约为 14B 的 $\approx$1/9（4B 约为 1/3.5），
并支持动态选择——简单子任务使用 1.5B Executor，复杂子任务调用 4B Executor——
对于高频故障监控（每分钟数千事件）至关重要。

\subsubsection*{技术优势}

将 MOLE1 与现有 LLM--SLM 协同及 SLM 蒸馏方法对比：

\begin{enumerate}[leftmargin=*, itemsep=4pt]
  \item \textbf{创新性。}
        现有 LLM--SLM 协同框架（FrugalGPT、RouteLLM）假设专家 SLM 已存在，
        完全聚焦于路由策略。现有 CoT 蒸馏方法（STaR、Distilling Step-by-Step）
        需要人工标注 CoT 或具备领域知识的教师模型。
        MOLE1 首次填补了冷启动空白：\emph{无需人工标注、无需领域感知教师}，
        仅以历史记录中已有的标准答案作为验证器，实现专家 SLM 的自动化生产。
        这填补了两条研究脉络中的关键盲点。
  \item \textbf{可部署性。}
        三循环设计在操作上高度透明：
        （Cycle 1）收集历史记录、运行教师模型采样、按答案匹配过滤；
        （Cycle 2）SFT+LoRA 训练初始 SLM；
        （Cycle 3）以操作匹配为奖励运行 AIOps-GRPO；
        （Bootstrap）用当前 SLM 扩充数据集并触发下一迭代；
        最终封装为 MCP 工具端点注册至 Planner 层。
        任何阿里巴巴 SRE 团队均可在数天内将流水线应用于新 AIOps 子场景，
        无需人工标注预算，仅需配置奖励权重与数据路径。
  \item \textbf{有效性。}
        领域专属 CoT 训练产出的 SLM，在窄领域任务上能与 GPT-4 级 LLM 持平甚至超越，
        尽管参数量相差数量级——这是蒸馏文献 \citep{dss,ddk} 中的成熟结论，
        MOLE1 将其推广至冷启动答案验证场景。
        "专业化优于泛化"现象的实证与理论分析将作为项目成果的一部分输出。
  \item \textbf{高效性。}
        MOLE1 标注开销为零，所需算力仅为训练或调用通用 LLM 的一小部分。
        推理阶段收益立竿见影：本地 SLM 执行（$<$100\,ms）消除了高频 AIOps
        监控任务的云端 API 成本与延迟，流水线开销远小于其替代的 LLM API 费用。
  \item \textbf{通用性。}
        MOLE1 仅需具有可自动验证答案的历史 $\langle$输入, 输出$\rangle$ 对——
        这一条件在工业界广泛成立（客服工单分流、代码审查分类、供应链异常检测）。
        设计完成后，可无需领域专属再工程即扩展至阿里巴巴多条业务线，
        成为广泛可复用的冷启动资产，远超本项目所演示的 AIOps 场景。
\end{enumerate}

\subsubsection*{讨论：为何选择 MOLE1，而非其他可行方案}

以下逐一分析批阅者可能想到的四类替代方案，并说明它们与 MOLE1 的关系或局限。

\textbf{方案一：仅用 RLVR/GRPO，跳过 Cycle 1 SFT 直接强化学习。}
将 GRPO 作为\emph{唯一}训练手段（纯自博弈强化学习，无 SFT 初始化）已在数学、
编程等任务展示出强大的 CoT 涌现能力。然而，纯 GRPO 有一个硬性前提：
策略模型在零/少样本条件下须已具备\emph{正概率}生成正确答案的能力，
才能产出任何正奖励 rollout 以驱动学习。
在冷启动 AIOps 场景中，1.5B/4B 目标 SLM 对新故障子场景的零样本成功率接近零，
纯 GRPO 与 STaR 面临同样的启动死锁。
此外，超参数（KL 系数、奖励权重、温度）的敏感性使纯 GRPO 在每次新子场景上
频繁调优的工程成本较高。

值得强调的是，MOLE1 \emph{并不放弃} GRPO——
Cycle 3 的 AIOps-GRPO 正是在 Cycle 2 SFT 之后施加的策略优化，
与原型系统 Logos 分子科学管道的 M-GRPO 阶段同构。
Logos 的消融实验表明，GRPO 阶段贡献了最大的精度提升。
MOLE1 在 AIOps 中保留了这一设计，但将奖励函数从
（需要 RDKit 等领域验证器的）化学合法性信号，
替换为操作序列精确匹配（$\mathcal{R}_{\mathrm{EM}}$）与结构化格式信号——
两者均无需任何额外工具，直接复用 Cycle 1 的验证函数，显著降低了部署门槛。
本方案讨论的是"\emph{是否}用 GRPO"，而 MOLE1 的答案是肯定的，
争议仅在于以何种奖励设计用之。

\textbf{方案二：RAG + 提示工程。}
检索增强生成（RAG）通过在推理时检索相关历史案例来提升 LLM 准确率，
无需训练，部署快速。然而，RAG 的核心局限在于：它\emph{不生产可本地部署的 SLM}。
每次查询仍须调用云端 LLM（延迟 500\,ms--2\,s），在高频故障监控场景
（每分钟数千事件）下既不可承受也不可靠。RAG 可作为 MOLE1 第一阶段
（数据采集）的辅助手段，但无法替代本项目的核心目标：产出一个延迟 $<$100\,ms
的本地专家 SLM。

\textbf{方案三：标准知识蒸馏（输出分布对齐）。}
标准 KD（MiniLLM \citep{minillm}、DDK \citep{ddk}）通过对齐学生模型与教师模型
的输出分布来压缩模型，技术成熟。但其前提是\emph{具备领域感知能力的教师模型}——
教师需能在目标领域生成合理输出，以提供有效的软标签监督信号。
冷启动场景下，通用 LLM（如 Qwen3-72B）可以生成输出，但其在领域专属任务上
的输出分布本身就是待优化对象：让学生模仿一个尚未专业化的教师，
本质上是在用未经验证的知识训练学生。MOLE1 的关键突破在于，
以\emph{已知标准答案}作为外部验证器，仅保留通向正确答案的推理链，
从而将通用 LLM 的生成能力转化为\emph{经验证的}领域特定训练数据，
规避了"教师本身不专业"的问题。

\textbf{方案四：STaR 自蒸馏。}
STaR \citep{star} 通过令模型自我采样推理链、按答案正确性过滤、再用过滤后数据微调自身，
实现无监督 CoT 涌现，在数学等任务上效果显著。
MOLE1 的答案验证过滤机制在形式上与 STaR 的过滤步骤相同，
但两者存在一项\emph{根本性的前提假设分歧}：
STaR 要求目标小模型在零样本或少样本条件下以\emph{非平凡正概率}自解任务，
即学生模型本身已能偶尔给出正确答案，才能产出任何通过过滤的样本。
而在冷启动 AIOps 场景中，1.5B/4B 目标 SLM 对新引入的子场景
（如新型服务网格故障或未见过的微服务交互模式）根本不具备解题能力——
若它已具备，该场景就不再是冷启动问题，本课题的研究前提也随之消失。
MOLE1 正是针对"学生无法自解"这一严格冷启动条件而设计：
引入任务可行的\emph{外部最小教师}（而非更大的 72B 全量 LLM，
后者成本高出约 $5{\times}$--$10{\times}$），打破自蒸馏的自引导死锁。
此外，如前所述，当过滤率为零时（STaR 的训练数据完全为空），
MOLE1 的 Planner 层提供了系统级降级路由，而 STaR 在此处无任何回退机制。

\textbf{综合结论。}
上述四类方案各有其适用场景，但均不能在冷启动约束下同时满足：
(i) 无标注数据要求、(ii) 产出可本地部署的 SLM、
(iii) 不依赖领域感知教师、(iv) 满足工业级延迟要求、
(v) 在目标小模型自解能力为零的严格冷启动条件下仍可运行。
MOLE1 是在这五项约束下唯一可行的方案，且其完整三循环机制
（Cycle 1 答案验证 CoT 合成 + Cycle 2 SFT + Cycle 3 领域聚焦 GRPO + Bootstrap 自举）
已在 Logos 分子科学原型 \citep{logos} 中得到量化验证：
专域训练的 4B 模型在结构准确率上达到 EM = 0.5588（ChEBI-20），
超越规模远大于其的通用 LLM（GPT-5 EM = 0.2488）。
Logos 消融实验进一步表明，GRPO 阶段（Cycle 3）贡献了最大的精度提升（EM 从 $\sim$0.20 提升至 $\sim$0.34），
证明了完整迭代训练策略的有效性，且该有效性与具体领域无关。

\section{交付成果}

本项目承诺如下最终交付成果。

\begin{enumerate}[leftmargin=*, itemsep=4pt]
  \item \textbf{算法原型系统。}
        完整的 MOLE1 三循环迭代 SLM 生产流水线：
        （Cycle 1）数据采集模块、CoT 候选生成与答案验证过滤器；
        （Cycle 2）SFT + LoRA 监督训练模块；
        （Cycle 3）AIOps-GRPO 策略优化模块（含奖励函数实现）；
        Bootstrap 自举回路（$\pi_t$ 对失败样本采样 $\to$ 过滤 $\to$ 写回 $\mathcal{D}^*$）；
        及 MCP 工具端点打包工具。
        源代码在 GitHub 开源发布；提供完整可复现实验环境。
  \item \textbf{MCP 合规 AIOps 工具端点套件。}
        针对 2--3 个 AIOps 子场景（数据库层故障、网络分区事件、容器资源耗尽）
        打包的 SLM Executor 原型，含 Planner 层集成文档与接入指南。
  \item \textbf{设计文档。}
        提交 1 份总体设计文档（系统架构与流水线规格）及 1 份详细设计文档
        （算法、评估方案与部署指南）。
  \item \textbf{论文目标。}
        投稿论文 2 篇至 CCF-A 或同等顶级会议（如 NeurIPS、ICML、ACL、ICSE、FSE），
        涵盖 MOLE1 方法与 AIOps 评估结果，争取于项目结题前至少 1 篇完成录用。
  \item \textbf{专利目标。}
        就冷启动 CoT 合成与 MCP 端点封装提交发明专利申请 1--2 项（国内或国际）。
  \item \textbf{技术指标。}
        与现有 AIOps 方案对比：MOLE1 生产的 SLM 修复操作准确率在 GPT-4 级 LLM
        的 5\% 以内，单次推理延迟降低 $>$10$\times$（从 $\sim$1\,s 降至 $<$100\,ms），
        单查询 token 成本降至零（本地推理）。
  \item \textbf{业务目标。}
        系统在 2 个以上 AIOps 子场景完成原型部署与性能验证，
        并有文档记录相对仅 LLM 基线的延迟与准确率提升。
  \item \textbf{项目验收报告。}
        项目结束时提交 1 份最终验收测试报告。
  \item \textbf{学生联合培养与实习。}
        联合培养参与本项目的研究生 2--3 名；在项目期间提供赴阿里巴巴
        SRE 团队实习机会（预计第 4--9 月，以阿里巴巴排期为准）。
\end{enumerate}
