% ──────────────────────────────────────────────────────────────────────────────
\subsection{项目标题}

\textbf{基于大模型冷启动蒸馏的专家小智能体自动化生产——面向 AIOps 大小模型协同故障应急}

% ──────────────────────────────────────────────────────────────────────────────
\subsection{背景与意义}

\textbf{问题建模：AIOps 故障应急的大小模型协同优化。}
阿里巴巴等大规模互联网平台须保障近乎零停机时间。
以电商大促为例，单次高峰期可触发数万条告警，人工 triage 每条平均耗时 5--15 分钟。%
\footnote{类似规模见 \textcite{aiops}。}
设故障事件 $\mathcal{E} = (\mathbf{x},\mathbf{a}^{*})$，
其中 $\mathbf{x} = (\text{告警序列},\text{日志片段})$ 为观测，
$\mathbf{a}^{*} = (o_1,\ldots,o_m)$ 为历史修复操作序列（标准答案）。
高频故障响应的目标是在满足修复正确率约束的前提下最小化端到端响应延迟：
\begin{equation}
  \min_{\pi_L,\,\{\pi_S^{(k)}\}_{k=1}^K}
  \;\mathbb{E}_{\mathcal{E}}\!\left[
    \tau_L\!\left(\pi_L,\mathbf{x}\right)
    +\max_k\,\tau_S^{(k)}\!\left(\pi_S^{(k)},s_k\right)
  \right]
  \quad\text{s.t.}\quad
  \mathbb{E}_{\mathcal{E}}\!\left[\mathbf{1}[\hat{\mathbf{a}}=\mathbf{a}^{*}]\right]\geq 1-\epsilon
  \label{eq:obj}
\end{equation}
其中 $\pi_L$ 为 Planner 大模型策略（将 $\mathbf{x}$ 分解为子任务集合 $\{s_k\}$ 并路由），
$\pi_S^{(k)}$ 为第 $k$ 类子场景的专家小模型策略，
$\hat{\mathbf{a}}$ 为各子任务输出的聚合修复序列，
$\tau_L \!\in\! [500\text{ms},2\text{s}]$、$\tau_S^{(k)}\!<\!100\text{ms}$
分别为大、小模型推理延迟。
课题一（大小模型协同）与课题十（AIOps）在此优化目标中天然交汇：
$\pi_L$ 依赖大模型的广度以泛化至新颖多步故障，
$\pi_S^{(k)}$ 则以极低延迟实现窄域深度。

\textbf{冷启动困境：专家小模型从何而来？}
式~\eqref{eq:obj} 的联合优化隐含一个前提——所有 $\pi_S^{(k)}$ \emph{均已存在}。
现有协同框架（FrugalGPT、RouteLLM）仅优化路由策略，同样依赖此假设，
却未回答更根本的问题：对\emph{新出现的}第 $k'$ 类 AIOps 子场景，$\pi_S^{(k')}$ 从何而来？
现有 SLM 生产方法——基于人工标注 CoT 的监督微调 \parencite{dss}、
知识蒸馏 \parencite{minillm,ddk} 或自举过滤 \parencite{star}——均依赖以下至少一项：
(a)~人工标注 CoT；(b)~具备领域知识的教师模型；(c)~已有非平凡解题能力的学生模型。
而在\emph{冷启动}场景——新 AIOps 子场景的常态——上述资源均不可得（图~\ref{fig:gap}）。
式~\eqref{eq:obj} 因缺失 $\pi_S^{(k')}$ 而无法求解，
整个大小模型协同系统陷入僵局。

\textbf{核心问题。}
如何在零人工标注、无领域感知教师的严格冷启动条件下，
仅从原始 $\langle\mathbf{x}, \mathbf{a}^{*}\rangle$ 对自动生产 $\pi_S^{(k')}$，
从而使式~\eqref{eq:obj} 的联合优化得以完整求解？
规则引擎无法泛化至新故障类型，ML 分类器缺乏可解释推理步骤，
RAG 每次仍须调用云端 LLM 而无法消除延迟瓶颈——均未解决这一研究空白。
本项目拟设计并验证一套全自动冷启动 SLM 生产流水线，
使协同优化目标式~\eqref{eq:obj} 在新子场景上持续可达。

\textbf{AIOps 是理想验证场景。}
三点使之成为最佳选择：
运维数据库中积累的 $\langle$告警+日志, 修复操作$\rangle$ 对天然满足所需格式，无需额外标注；
修复有效性既可通过历史记录字符串匹配验证，也可在故障回放沙箱中以\emph{执行结果}（内存峰值恢复、服务心跳、错误率下降、接口延迟）直接衡量，验证信号远比通用文本任务丰富；
本地 SLM 的 $<$100\,ms 延迟与云端 API 的 500\,ms--2\,s 形成直接量化对比，指标清晰可复现。

\subsection{研究目标}

本项目设定四项紧密关联的研究目标：

\begin{enumerate}[leftmargin=*, label=\textbf{O\arabic*.}, itemsep=3pt]
  \item \textbf{冷启动专家 SLM 生产框架。}
        设计并验证一套三循环迭代流水线，仅从原始 $\langle$输入, 输出$\rangle$ 对出发，
        经由答案验证 CoT 合成（Cycle 1）、监督推理训练（Cycle 2）、
        领域聚焦策略优化（Cycle 3）及自举（Bootstrap）回路，
        实现冷启动条件下专家 SLM 的全自动化生产。
  \item \textbf{AIOps 场景系统验证。}
        在阿里巴巴运维数据集及公开 AIOps 基准 \parencite{aiops} 上验证所提流水线，
        证明本地专家 SLM 在故障诊断与修复准确率上能达到或接近通用大模型的水准。
  \item \textbf{专业化机制实证分析。}
        通过消融实验（变量为模型规模、CoT 过滤率）
        及表示空间分析，对"优质 CoT 训练使小模型在窄域任务上超越大模型"现象进行实证论证。
  \item \textbf{跨域适用性初探（探索性）。}
        在资源允许时，将流水线应用于 AIOps 以外的公开数据集，
        初步验证本方法的领域无关性；否则以消融实验形式作为后续工作铺垫。
\end{enumerate}

\noindent\textbf{工程交付物（非研究目标）：}
将训练好的 SLM 打包为专家小智能体，集成至 Planner 层，负责有效处理计划大模型分发的子任务，
作为研究成果向生产落地的工程出口，不构成独立研究贡献。

\noindent\textbf{量化目标：}
SLM 修复准确率与 GPT-4 级 LLM 差距 $\leq$5\%，推理延迟降低 $>$10$\times$（$<$100\,ms），
token 成本降至零（本地推理）；投稿 CCF-A 论文 2 篇（争取 1 篇结题前录用）；申请发明专利 1--2 项；
2 个以上 AIOps 子场景完成原型部署与性能验证。

\subsection{研究方法}

\subsubsection*{核心思路：三循环迭代答案验证 CoT 合成}

本项目拟设计一套\textbf{三循环迭代 CoT 合成流水线}，领域无关，
\emph{无需任何人工标注或具备领域知识的教师模型}，
将原始 $\langle$输入, 输出$\rangle$ 对转化为具有推理能力的专家 SLM。
系统由三个相互依存的训练循环构成，并配有一个自举（Bootstrap）回路：

{\small
\begin{alignat}{2}
\text{Cycle 1:}&\quad
  \underbrace{\langle q,\, a \rangle}_{\text{冷启动数据}}
  &\;\xrightarrow{\;\text{最小教师},\;k\text{ 采样 + 答案过滤}\;}&\;
  \underbrace{\mathcal{D}^*}_{\text{验证三元组集}} \notag\\[4pt]
\text{Cycle 2:}&\quad
  \mathcal{D}^*
  &\;\xrightarrow{\;\text{SFT+LoRA}\;}&\;
  \underbrace{\pi_0}_{\text{初始 SLM}} \notag\\[4pt]
\text{Cycle 3:}&\quad
  \pi_0
  &\;\xrightarrow{\;\text{AIOps-GRPO}\;}&\;
  \underbrace{\pi^*}_{\text{专家 SLM}} \notag\\[4pt]
\text{Bootstrap:}&\quad
  \pi_t\bigl(\mathcal{D}_\emptyset\bigr)
  &\;\xrightarrow{\;\text{过滤}\;}\;
  \mathcal{D}^*_{\text{new}} \hookrightarrow \mathcal{D}^*
  &\;\xrightarrow{\;\text{触发下一轮 Cycle 1}\;}\;
  \pi_{t+1} \notag
\end{alignat}}

\noindent 其中 $q$ = 查询，$a$ = 标准答案，
$\mathcal{D}_\emptyset$ = 本轮过滤率为零的失败样本集，$\pi_t$ = 第 $t$ 轮训练所得 SLM。
这一设计的关键在于：以历史记录中\emph{已有}的标准答案作为唯一外部验证信号，
无需引入额外人工监督，即可自动筛选通向正确答案的推理链，从而将通用模型的生成能力
转化为经验证的领域特定训练数据。
Logos \parencite{logos} 在科学计算领域的实验已表明：以答案验证自动筛选的优质 CoT 样本对小模型进行训练，
可使其性能大幅超越原有水平，甚至能完成此前完全无法解决的任务——
\emph{提升小模型推理能力}才是核心目标，答案验证仅是自动筛选高质量训练数据的手段。
本项目将这一机制迁移至 AIOps 冷启动场景，系统探索其适用条件与奖励信号设计。

\subsubsection*{三循环训练流程}

图~\ref{fig:pipeline} 展示了完整的三循环流水线。

\begin{figure}[H]
\centering
\input{fig_pipeline.tikz}
\caption{三循环迭代流水线：Cycle 1 产出验证数据集，Cycle 2 SFT 得到初始 SLM，
Cycle 3 策略优化强化推理质量，Bootstrap 回路驱动数据与模型协同进化。}
\label{fig:pipeline}
\end{figure}

\begin{description}[leftmargin=0pt, itemsep=6pt]

  \item[Cycle 1——数据采集与验证 CoT 合成。]
        从运维日志中收集 $\langle$告警+日志, 修复操作$\rangle$ 对，
        无需任何人工标注；数据来源于现有故障管理系统。

        针对每条查询，由\emph{任务可行的最小外部教师模型}
        （温度采样 $\tau{=}0.8$，top-$p{=}0.95$）生成 $k$ 条候选推理链。
        冷启动条件下目标 SLM 尚不具备解题能力，故须借助外部教师启动流程；
        选用最小可用模型（而非 72B 全量模型）可将生成成本降低约 $5{\times}$--$10{\times}$。

        对每条候选推理链，提取末尾预测操作序列 $\hat{a}_j = \mathrm{ans}(c_j)$
        与标准答案 $a = (o_1, \ldots, o_m)$ 进行集合等价匹配（规范化后）：
        \[
          \mathcal{D}^* = \bigl\{\,(q,\, c_j,\, a)\;\bigm|\;
            \mathrm{ans}(c_j) \equiv a,\; j = 1,\ldots,k\,\bigr\}
        \]
        操作序列的结构化性质（每个 $o_i$ 为 \texttt{restart\_service}、
        \texttt{scale\_pod} 等运维指令）保证匹配判断\emph{完全自动化}，无需领域知识。
        涉及操作员失误或标签模糊的记录通过轻量异常过滤器排除；
        保留率通常在 30--70\% 区间（$k{=}8$），足以为下一阶段提供高质量训练数据。

  \item[Cycle 2——有监督推理训练（SFT + LoRA）。]
        将 $\mathcal{D}^*$ 中的验证三元组 $\langle q, c^*, a \rangle$ 用于
        对紧凑型 SLM（简单子任务选用 1.5B，复杂子任务选用 4B）
        进行带 LoRA 的监督微调，最大化 CoT 与答案的联合自回归似然。
        损失函数对推理链过短（$|c| < \gamma$）的样本降权，
        防止模型旁路推理步骤直接输出答案。
        此阶段产出初始 SLM $\pi_0$，
        它已能遵循任务格式并输出合理推理链，但准确率尚待进一步强化。

  \item[Cycle 3——领域聚焦策略优化（AIOps-GRPO）。]
        仅靠 SFT 无法保证推理链导向正确操作——SFT 拟合的是采样分布，
        而非最优推理策略。
        对 $\pi_0$ 施加群体相对策略优化 \parencite{onpolicy}：
        每条 prompt 生成 8 条 rollout，按奖励计算组内相对优势并更新策略参数。

        \textbf{AIOps 奖励设计。}
        与通用 NLP 任务不同，AIOps 操作均可在故障回放沙箱中真实执行，
        因此奖励信号可分为\textbf{轻量符号层}（无需执行）与\textbf{执行结果层}（沙箱验证）两层：

        \smallskip
        \noindent\textit{轻量符号层（每条 rollout 即时计算）：}
        \begin{itemize}[itemsep=1pt]
          \item $\mathcal{R}_{\mathrm{format}}$：输出符合结构化操作列表格式；
          \item $\mathcal{R}_{\mathrm{CoT}}$：推理链长度超过最小阈值（激励实质性推理）；
          \item $\mathcal{R}_{\mathrm{EM}}$：操作序列与标准答案精确匹配；
          \item $\mathcal{R}_{\mathrm{soft}}$：操作类型匹配但参数有差异时给予部分分；
          \item $\mathcal{R}_{\mathrm{len}}$（负项）：对过长推理链的软惩罚。
        \end{itemize}

        \noindent\textit{执行结果层（沙箱异步回放，批量写回奖励缓冲）：}
        \begin{itemize}[itemsep=1pt]
          \item $\mathcal{R}_{\mathrm{mem}}$：执行修复操作后容器内存峰值是否回落至正常阈值，
                对"内存泄漏/OOM"类故障提供比字符串匹配更直接的验证；
          \item $\mathcal{R}_{\mathrm{svc}}$：服务重启后健康检查（HTTP/gRPC 心跳）是否在
                超时窗口内通过（二值信号）；
          \item $\mathcal{R}_{\mathrm{err}}$：修复后滚动窗口内应用错误率下降比例
                （$\Delta\,\mathrm{err} / \mathrm{err}_{\mathrm{pre}}$）；
          \item $\mathcal{R}_{\mathrm{lat}}$：关键接口 p99 延迟相对修复前的恢复程度，
                用于网络分区或慢查询类故障的连续奖励。
        \end{itemize}

        总奖励 $\mathcal{R}_{\mathrm{total}} = \lambda_s \mathcal{R}_{\mathrm{sym}} + \lambda_e \mathcal{R}_{\mathrm{exec}}$，
        其中 $\mathcal{R}_{\mathrm{sym}}$ 为符号层加权和，$\mathcal{R}_{\mathrm{exec}}$ 为执行层加权和。
        训练早期以符号层为主（执行沙箱尚未就绪），中后期逐步增大 $\lambda_e$；
        KL 散度惩罚限制策略偏离初始点以保持训练稳定。
        执行层奖励的引入是 AIOps 场景区别于通用文本任务的关键优势：
        系统可以直接观测操作是否"真的修好了故障"，而非仅判断输出字符串是否与历史记录一致。

  \item[Bootstrap 回路——自举数据扩充。]
        当某子任务在 Cycle 1 过滤率为零（$\mathcal{D}^* = \emptyset$）时，
        训练数据完全为空，流水线陷入死锁。
        迭代架构天然处理这一情形：
        用\emph{当前已训练的 SLM}（$\pi_t$，而非外部教师）对失败样本重新采样，
        经答案验证过滤后写回 $\mathcal{D}^*$，触发下一轮迭代，
        实现数据集与模型能力的协同进化。
        在积累足够 Bootstrap 样本前，该子任务请求暂由 Planner LLM 直接处理，
        维持系统可用性而不产出空训练集。

\end{description}

\subsubsection*{实验设计与评估方案}

\textbf{数据集。}
采用两类数据结合：
(a)~来自阿里巴巴运维日志的内部 AIOps 数据集（已与 SRE 团队初步确认数据访问意向，
数据共享协议正在商定中），预估规模不低于 10,000 条故障事件，
以 8:1:1 比例切分训练/验证/测试集；
(b)~公开可用的 AIOps-2022 基准 \parencite{aiops}（保障可复现性）。
实验覆盖三个子场景：数据库层故障、网络分区事件和容器资源耗尽。
CoT 候选生成阶段每条查询采样 $k{=}8$ 条推理链。

\textbf{对比基线。}
\begin{itemize}[itemsep=1pt]
  \item \textbf{仅 LLM}：通过 API 调用 72B，无本地 SLM。
  \item \textbf{SLM（无 CoT）}：SLM 仅在 $\langle$输入, 输出$\rangle$ 对上微调，无推理链。
  \item \textbf{SLM + 教师 CoT}：SLM 在由 72B 作为教师生成的 CoT 上微调（oracle 上界）。
  \item \textbf{本方案}：SLM 在答案验证 CoT 上微调，教师仅为任务可行的最小模型。
\end{itemize}

\textbf{评估指标。}
\emph{符号层}：修复操作准确率（top-1 命中率）、推理延迟（p50/p99）、单查询计算成本（FLOPs / API token）。
\emph{执行层}：沙箱回放后内存峰值恢复率、服务心跳通过率、滚动窗口错误率下降比、p99 延迟恢复比。
统计显著性通过自助重采样（$n{=}1000$）评估。

\subsubsection*{系统架构：Planner + 专家小智能体}

图~\ref{fig:arch} 展示了整体部署架构。

\begin{figure}[H]

云端 LLM（72B 或同等模型）充当 \textbf{Planner}，将故障事件分解为子任务并路由至对应的\textbf{专家小智能体}（SLM）；
每个小智能体专为单一 AIOps 子场景训练（如数据库慢查询诊断、网络分区检测、容器 OOM 分诊），
负责有效处理 Planner 分发的任务，本地推理延迟 $<$100\,ms。
Planner--专家小智能体分层设计同时保留了大模型的全局语义广度与小模型的窄域推理深度 \parencite{dss,ddk}，
端到端系统可在秒级内响应高频告警。
本项目将通过消融实验与表示空间分析对"优质 CoT 训练使 SLM 在窄域任务上超越通用大模型"这一机制进行系统验证。

\section{交付成果}

本项目承诺如下最终交付成果。

\begin{enumerate}[leftmargin=*, itemsep=4pt]
  \item \textbf{算法原型系统。}
        完整的三循环迭代 SLM 生产流水线：
        数据采集模块、CoT 候选生成与答案验证过滤器（Cycle 1）；
        SFT + LoRA 监督训练模块（Cycle 2）；
        AIOps-GRPO 策略优化模块，含奖励函数实现（Cycle 3）；Bootstrap 自举回路。
        源代码在 GitHub 开源发布，提供完整可复现实验环境。
  \item \textbf{AIOps 专家小智能体套件。}
        针对 2--3 个 AIOps 子场景（数据库层故障、网络分区事件、容器资源耗尽）
        打包的专家小智能体原型，含 Planner 层集成文档与接入指南。
  \item \textbf{设计文档。}
        提交 1 份总体设计文档（系统架构与流水线规格）及 1 份详细设计文档
        （算法、评估方案与部署指南），分别于 M1、M3 里程碑交付。
  \item \textbf{论文目标。}
        投稿论文 2 篇至 CCF-A 或同等顶级会议（如 NeurIPS、ICML、ACL、ICSE、FSE），
        涵盖所提方法与 AIOps 评估结果，争取于项目结题前至少 1 篇完成录用。
  \item \textbf{专利目标。}
        就冷启动 CoT 合成与专家小智能体生产提交发明专利申请 1--2 项（国内或国际）。
  \item \textbf{技术指标。}
        所产出 SLM 修复操作准确率在 GPT-4 级 LLM 的 5\% 以内，
        单次推理延迟降低 $>$10$\times$（从 $\sim$1\,s 降至 $<$100\,ms），
        单查询 token 成本降至零（本地推理）。
  \item \textbf{业务目标。}
        系统在 2 个以上 AIOps 子场景完成原型部署与性能验证，
        并有文档记录相对仅 LLM 基线的延迟与准确率提升。
  \item \textbf{项目验收报告。}
        项目结束时提交 1 份最终验收测试报告。
  \item \textbf{学生联合培养与实习。}
        联合培养参与本项目的研究生 2--3 名；在项目期间提供赴阿里巴巴
        SRE 团队实习机会（预计第 4--9 月，以阿里巴巴排期为准）。
\end{enumerate}
